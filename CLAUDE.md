# CLAUDE.md

Guidance for Claude Code (claude.ai/code) working in this repository.

## What this is

**Crash Bet** — a betting game about reading physics, built on a deterministic crash sim and a seeded procedural low-poly world. Every round deals a generated traffic scene ten seconds from disaster, the player bets on what the chaos will do, and the sim settles every bet with zero ambiguity. Vehicles and property only — no people, ever.

Vanilla JS + Three.js r169 + Rapier 3D 0.19, both vendored (`libs/`, import map in `index.html`). No build step, no package.json, no framework, fully offline, installs as a PWA. **The game is feature-complete and not deployed.**

**App shell**: boots to a main menu gated behind a boot preloader (physics + Rapier wasm + fx pools + showroom warm up behind a progress bar; `preloaded` flips at 100 %). Menu: **Continue/Resume** (campaign) · **Daily** (one date-derived scene a day) · **Stats** (record + 16 achievements) · **Custom seed** (Exhibition) · **Garage** (showroom of all 280 models) · **How to play** · **Crash test** (8 seeded set-piece wrecks) · **New run**. In game: hamburger → pause menu (Resume / Settings: environment · quality · camera · volume · motion / Fullscreen / Leave). Dev URL params: `?scene=<seed>~<d>`, `?crash=N`, `?play=1`, `?smoke=1`, `?simtest=1`, `?sheet=1` (vehicles) / `?sheet=scenery` (all 184 scenery models).

**The editor is mothballed, not deleted**: `js/editor.js` (spawn/gizmo/inspector/URL codec/worldgen UI) is intact but not imported, and its DOM was removed from `index.html` — re-wiring needs that DOM back plus `initEditor(ctx)`. Editor-era URL params (`car`/`prop`/`road`) are no longer parsed. Editor-era CSS is kept at the top of `css/style.css`; the game's styles are the marked section below it.

## Running & testing

- Serve over HTTP (ES modules will not load from `file://`): `run.bat`, or `npx http-server -p 5183 -c-1 .`. `.claude/launch.json` has a `garage` config for the browser preview.
- **Smoke test**: `?smoke=1` — builds every registry type × 4 seeds, logs `SMOKE DONE: N/N ok`. Run after changing `vehicles.js`, `families.js`, `parts.js`, `scenery.js` or `lib.js`. Currently **1104/1104** (92 vehicles + 184 scenery, × 4). It is a coverage count, not a pin — it moves whenever the registries grow.
- **Contact sheets**: `?sheet=1` tiles every vehicle, `?sheet=scenery` every scenery model, into one canvas at `window.__sheet`. Both run synchronously at boot so they work with rAF suspended. With 184 scenery models this is the only practical way to review a batch, and reviewing a batch is how backwards winding, floating parts and wrong-axis rotations get caught — every defect in the §1D batch was found this way and none of them failed the smoke test.
- **Determinism self-test**: `?simtest=1` — runs every `TEST_SCENARIOS` entry in physics.js twice (300 fixed steps), FNV-hashes every body transform per step (debris included) plus the final crumpled geometry. Run after changing `physics.js`, `deform.js`, `props.js`, `scenery.js`, `roads.js`, `worldgen.js` or `director.js`. Hashes are stable across node and browser (same wasm f32 math).
- **Pin gate**: `node tests/simpins.mjs` — `?simtest=1` only proves run-to-run stability; this proves the values never MOVED, which is the invariant that actually protects the sim. **Current pins (12)**: legacy `769824bd`/geo `f11ed37a` · extended `54489fae`/`20413f1b` · scenery `50fa7593`/`ec076bdc` · roads `4ac04ea9`/`ab859269` · deck `354b55b0`/`f0c5be44` · bridge `41c337c8`/`af80eb9` · water `ac116fb4`/`9d559754` · waterv2 `6ce9b062`/`aa4fcf44` · carnage `98c34be0`/`584ce83a` · director `e327af80`/`1592158f` · wxgrip `82c01590`/`b42a8bfb` · worldgen `e9050265`/`9baac2f5`. None may drift unless physics/deform intentionally changed — **update this line when they do**, and record why in the file next to the pin. `director` moved twice in Phase 2, both times deliberately and both recorded next to the pin: 2A (stubs ±6.3 → ±7.4, `asphalt_patch` and its `rDress` draw gone) and 2I/2J (traffic signals + cast 3-8 → 4-10). The second moved the GEOMETRY hash too, which is expected when the cast changes and is not evidence of a physics change. `worldgen` moved once, in 2G, when the suburb generator gained a rural fringe (barns/silos/orchard rows) — pure appended dressing, determinism intact. `director` moved a third time at the close gate (ledger #35): ambient cars at a signalized junction now need ROOM to stop before being seated at the bar, so one placed too close and too fast to pull up is culled rather than left to overshoot the box — cut pre-incident ambient grazes 0.50 % → 0.17 % of intersection scenes. **2E and 2F moved NO pin**: pin-1 stays `intersection`/`police` (no 2E topology reshuffles its rng draw to a new index, no 2F template deals on intersection), so the entire twelve-topology / eight-template expansion is provably physics-neutral.
- **Scrub gate**: `node tests/scrub.mjs` — the freeze scrub rewinds by re-simming from t0, so this asserts a scrub round-trip lands on the EXACT state a clean run reaches (per seed × difficulty, rewinding to arbitrary ticks including 0). A money-correctness gate, not a UI one: the slip settles against a recording whose incident starts at `INCIDENT_TICK`, so a sim resuming from a different state plays a scene the odds were never priced on.
- **Economy suite**: `node tests/economy.mjs` — payout flooring, bankroll floors, once-per-seed ledger, parlay caps, mid-round resume, the Exhibition rule, the daily seed (shared across profiles, one paying attempt, streak arithmetic across month/year boundaries), achievements (idempotent, Exhibition unlocks nothing, a throwing test cannot break settlement), plus a real scene → markets → settlement integration pass. Currently **56/56**.
- **Weather gate**: `node tests/weather.mjs` — weather never touches the sim, so no pin can catch a mistake in it. What this actually guards is the silent-fallback class: `WEIGHTS` falls back to `proving` on an unknown env id, so it asserts `ENVS` and `WEATHER_ENVS` agree **in both directions** (a preset added without a table would inherit a desert's weather, and nothing would say so — precisely how `city` went unnoticed). Plus descriptor invariants, roll determinism, grid staying clear, and ≥4 kinds per env because "every scene on a preset looks identical" is the problem weather exists to solve. Currently **42/42** (it scales with `ENVS` — 21 at 5 presets, 36 at 10; P2/2D added 6 grip assertions). The grip block guards the same silent-fallback class from the other end: `gripFor` returns 1 for anything it does not recognise, so a kind added to `KINDS` without a `grip` would quietly become a dry road. It also pins the two facts the opt-in's safety rests on — dry kinds are **exactly** 1 (a `clear` row of 0.999 would move every pin the moment a topology enabled grip) and grip falls monotonically along the rain ramp, because a tell the player cannot read is the bug. Run after touching `weather.js` or `ENVS`.
- **Scene sweep**: `node tests/sweep.mjs [seedsPerD]` (default 60 → 180 scenes) — deals across d 1/5/10, records to rest, reports coverage per topology × template. Hard defects: a car↔car hit before the incident tick, no markets, a missing headline pair. Quality *budgets* (rates, not per-scene defects): ran-to-cap ≤ 10 %, eventless ≤ 12 % — both are legitimate outcomes, so they are tracked with headroom rather than pretended away. The pre-incident check counts **car↔car only**: the recorder logs kerb and ground contacts as `hit` too, and clipping a kerb on the approach is not a choreography failure.
- **Monte Carlo gate**: `node tests/montecarlo.mjs [roundsPerD]` (default 100 × 10 ≈ 9 min) — bets every market to-win-$100 and asserts the realized house edge sits inside the per-difficulty band around `marginFor(d)`. Seeds are fixed (`mc…`), so **re-running the same size is deterministic and cannot tell you whether a band miss is noise** — vary `roundsPerD` to resample. Currently **10/10 bands, overall edge 4.59 %** over $983 k staked (P2 close gate, against CALIB regenerated over 900 scenes). The edge runs hotter than G4's 2.45 % because the fresh table's out-of-sample O/E sits at ~0.97 (house-favorable, the safe direction) — within the 900-scene calibration's ±0.04 noise, so no difficulty correction was added and every band still passes. Calibration seeds (`cal…`) and gate seeds (`mc…`) are disjoint, so the measurement is genuinely out-of-sample. Beware short runs: at 15 rounds/d the estimator's σ is ≈ 7 %, so a small sample "fails" on noise alone.
- **Odds calibration**: `node tools/calibrate.mjs [n]` prints the empirical CALIB table to paste into `js/markets.js`. Regenerate whenever director/physics changes move outcome statistics. It also prints, to stderr, observed/expected per difficulty band with a 95 % interval — that is the number the Monte Carlo actually tests (edge ≈ `1 − (1 − margin)·O/E`), and it is far more diagnostic than reading edge percentages afterwards.
- **Headless physics in node** (much faster than the browser loop): `node_modules/three/` is a gitignored shim so the whole chain imports in node. Import via `pathToFileURL`, build a `CrashSim`, call `stepOnce()`, assert on `body.translation()` / `hashState()`.
- Syntax check without a browser: copy a module to `.mjs` and run `node --check`.
- **Headless visual verification**: `window.__app` exposes `{ renderer, scene, camera, controls, REG, env, fitCamera, invalidate, startGame, leaveGame, setCamMode, fly, flyUpdate, showroom, startCrash, nextCrash, replayCrash, crash, crashFx, startScene, resumeRound, seekPreview, round, preloaded, pump, weather, rollWeather, veg }`. **`__app.pump(now)` runs one real frame** (camera tweens, sim, fx, HUD, render) with a synthetic clock — loop it with `now += 1000/60` to drive the game headlessly instead of re-implementing the loop. DOM overlays can be rasterized via SVG `foreignObject` with the stylesheet inlined (serialize with `XMLSerializer` — raw `innerHTML` is not valid XML).

## Architecture

Module chain (each imports the previous): `lib.js` → `parts.js` → `families.js` → `vehicles.js` → `main.js` (+ `names.js`). Sim: `deform.js` + `props.js` → `physics.js` → `main.js` (+ `env.js` + `fx.js`); `terrain.js` is pure and imported by `env.js` for the mesh and (since P2/2C) by `physics.js` for the drivable heightfield — both must sample the SAME `makeHeightField` or collider and landscape drift apart; `water.js` imports `lib.js` and is likewise owned entirely by `env.js` (env holds the lifecycle, water.js the look); `weather.js` imports only `lib.js` and is consumed by `main.js` **and by `director.js`**, which calls its two pure functions (`rollWeather`/`gripFor`) to put the scene's weather on the scenario — env.js still takes a weather *descriptor*, never the module; `vegetation.js` imports `lib.js` + `scenery.js` and is driven by `main.js` from `env.terrainField`. Game: `director.js` (imports roads + worldgen + weather's pure roll, NEVER physics) → `recorder.js` (imports physics + director + roads) → `main.js`; physics.js imports director.js only for its pinned `director` scenario. `signals.js` is pure (program + tick evaluation, no THREE, no rng at eval time) and is imported by `director.js` to generate, `physics.js` to obey and `main.js` to light the lamps. `markets.js` → `economy.js` → `betui.js`. `achievements.js` is pure and imports nothing. `editor.js` is dormant. Rapier is lazy-loaded (2.2 MB) during the boot preloader.

**Physics (`physics.js`) — determinism is the product.** Fixed 60 Hz timestep + accumulator; rendering interpolates and never feeds back. Zero `Math.random()` in sim code; world creation order is the backbone: ground → arena walls → roads → props → cars (array order). Chassis colliders are convex hulls from each slab's `userData.pt`, rescaled to a per-category target mass (`CAT_PHYS`) with a no-contact ballast collider lowering the COM. Wheels come from `userData.wheel` tags; duals dedupe, inline bikes get invisible outriggers, tracked vehicles get 4 virtual corner wheels. Raycast vehicle controller per car. Per-car tuning multipliers default to exactly 1.0 (`rest: null` → 0.12) so legacy scenarios hash bit-identically. Scenario shape `{ world: {gravity, arena, walls, water, terrain, weather}, roads, junctions, props, cars }`; "perfect reset" = rebuild from the scenario (~30 ms for 10 cars). **Incremental editing** (`replaceCar`/`appendCar`/…) touches one object for paused responsiveness and does NOT promise a deterministic world — that is why the editor flags `dirty` and Play/Reset/Step always start from a full `build()`. **Damage model**: `CAT_PHYS` carries per-category `sus`, `aDamp`, `crumpleK`, `wheelTough`; impacts charge nearby wheels, past `WHEEL_BENT_AT` a wheel steers crooked, past `WHEEL_DETACH_AT` it tears off into a real dynamic debris body inheriting `v + ω×r` (`hashState` folds debris in; any new debris path MUST register with `this.debris` or cleanup and hashing miss it). Six render-side hooks — `onImpact/onScrape/onGlass/onDetach/onSplash/onSunk` — fire inside `stepOnce`; **the only sanctioned write-back is `sim.speed`** (wall-clock pacing).

**Deformation (`deform.js`).** Non-indexed geometry, vertices weld by position (half-mm quantization), displacement applies per weld group so faces never tear. Depth scales with contact Δv, radial falloff + position-hash grain, accumulated crush clamped per group. Visual only — colliders stay rigid. Per-group zone stiffness (nose/tail crumple deep, cabin resists, roof gives in rollovers), work hardening, ridged crease noise + tangential shear, and two-stage glass (crack → shatter) driven by direct impact overlap plus inverse-square "frame shock" that pops windshields from bumper hits.

**Roads (`roads.js`).** Catmull-Rom spline roads (centripetal, so tight zig-zags never cusp). `buildRoad(spec)` sweeps asphalt + geometry lane markings, curbs, optional sidewalks and crosswalks, returning `{ group, shapes }` — shapes are curb box-collider recipes physics attaches to ONE fixed body at the origin (the group is never transformed). Zero randomness: segment counts derive from arc length. Strip winding rule: horizontals face up when `o0<o1`; the mirrored side negates offsets AND flips winding — pass offsets ascending for s=+1 or faces render invisible from above. Style bitfield: 1 = double yellow, 2 = sidewalks, 4 = crosswalks, 8 = bridge deck, **16 = rural verge, 32 = swept guardrail** (both P2/2B, both ignored on a deck).
**Road model (P2/2B).** Four additions, all sim-neutral except the two new opt-in bits (verified: styles 0/1/3/7 share one collider hash and 8/9/15 another, so a spec that does not set 16/32 emits the shape list it always did, and all 11 pins held).
- **A verge REPLACES the kerb and emits NO collider.** Every road in the game had a raised kerb, highways and mountain roads included — both wrong and a wall where the scene wants a run-off. `16` swaps it for a gravel shoulder falling to −0.05, so leaving the road is a run-off rather than a kerb strike. `32` sweeps a real guardrail (posts on arc-length stations, beam, colliders at `GRAIL_H`), because a barrier that does not contain a car is scenery — the existing `guardrail` scenery kind is a discrete 4 m object, this is the continuous run a mountain road needs. Verge+guardrail together gives the guardrail's colliders.
- **Lane count from width**: past 10.5 m a road gets interior dashes. A 12 m road with one centre line read as an absurdly wide two-lane. Visual only — director lane extraction is untouched, so no scene changes shape.
- **Tyre polish** down each wheel path, always on, derived from geometry like everything else here. Costs one material (so one merged mesh) per road and is what stops a long straight reading as one flat swatch at the grazing angles a dashcam actually sees.
- **Sweep caps raised 480 → 900 and 220 → 400** (ledger #27). Both bound at ~528 m and nothing reaches it: across 514 sampled roads the longest is 321.8 m (switchback), so the raise is a bit-level no-op *today* — the clamp was never active — while removing a silent coarsening trap for longer P2 topologies. The pins prove the no-op.
**Elevation** is opt-in and defaults to exactly 0, which is why every pre-elevation hash survived it: Catmull-Rom interpolates components independently so all-zero y gives exactly 0 out, and every height is an offset ABOVE the frame's surface (`f.y + y0`) so `+ 0` is a bit-level no-op — do not "simplify" that to absolute heights. Tangents normalise by HORIZONTAL length, so the ribbon never narrows as it climbs. Curb colliders keep the yaw-only quaternion on flat roads and only pay for the pitched basis when `isElevated(spec)` — routing flat roads through a matrix→quaternion conversion would perturb the last bits. **Elevated runs also emit a driving-surface slab** (opt-in on `elev`): without it the deck is decoration and a car mid-span falls straight through. Its longitudinal overlap is deliberately much smaller than the curbs' — consecutive slabs pitch to their own tangent, so a generous overlap leaves the next slab's leading edge proud, a lip that trips a wheel at speed. Nothing drives on a curb; something drives on this.
**Substructure (1G)** is drawn when `deck && elev` and is **visual only — nothing reaches `shapes`**, so no collider changes and no pin can move (cars are already contained by the parapets and held by the deck slab). Edge girders, pier bents on ~17 m arc-length stations wherever the deck clears `PIER_MIN`, pier caps, and abutments walked in from each open end to where the deck first lifts clear of grade. Two things make it simple: **columns run to `PIER_FOOT` well below 0**, so on land the ground mesh buries the footing and over a punched water basin the same column carries on down and founds itself in the bed — one number, both cases, and roads.js never learns what water is; and the **girders sweep the full length** rather than just the elevated stretch, buried at grade by the same trick.
**Junctions (`buildJunction`, P2/2A).** `{ x, z, arms: [{a, w}], reach, fillet, style }` → `{ group, shapes }` where **`shapes` is always empty** — a junction is flat asphalt on the y-0 plane, the ground slab already holds cars up, and that is why adding them moved no pin (verified: stripping `junctions` from the `director` scenario leaves all 300 tick hashes bit-identical at 510 colliders either way). They live on `scenario.junctions`, are built in `physics.js` between roads and props purely to share the scene root, and main.js must include them in **both** the wetness pass and the merge/freeze pass — a dry square in the middle of a wet intersection is the one place the camera always looks. Style bits: 1 stop bars · 2 crosswalks · 4 keep-clear box · 8 turn arrows.
- **Everything is drawn 2 mm BELOW the equivalent road layer**, and the whole design hangs off that: a stub may overlap the junction freely without z-fighting, so the polygon never has to be tangent to anything and arms can be any width at any angle. It is the trick the old `asphalt_patch` prop used; the patch's mistake was being a fixed 13 × 13 m square rather than a shape derived from the arms actually meeting. Stubs now START just inside `reach` so their blunt ends are covered (measured: `road@y0.022` over `JUNCTION@y0.020`).
- **Corner fillets round a REFLEX corner, so they ADD asphalt.** Centre solves `(X−P)·a = (X−P)·b = r` for unit outward normals, giving `X = P + r(a+b)/(1+a·b)`; both tangent points then slide outward along their edges *linearly* in r, so the largest r that still fits inside `reach` has a closed form — no search, nothing engine-dependent.
- **Markings key off the CONFLICT extent, never off `reach`.** A cross road of half-width h meeting an arm at angle θ crosses its axis at `h/|sin θ|`, so an opposite arm contributes nothing (a T-junction gets its bar at the stem, not pushed out by the far arm) and a skew arm correctly contributes more than h. The keep-clear box is a **rectangle** from the same expression, not a square of `hmax` — a square buried its own corners in the crosswalk bars.
- The approach lane is offsets `0..+h` along `n`: traffic inbound on an arm travels `−d`, whose right-hand side is `+n` under the left-normal convention. Mirror it and every stop bar and arrow paints in the oncoming lane, which no test reports. Verified all four arms against the director's own lane table.
- **Winding, for the third time in this project** (after the strip rule above and terrain.js's spokes) — and a fourth in `pushRing`, which needs the OPPOSITE reversal to `pushLocal`: a world ring parameterised by θ has radial × tangential = −Y, while an arm's `(d, n)` frame has d × n = +Y. The two rules look contradictory and are not. Copying `hexa`'s quad order is necessary but NOT sufficient — it only yields outward faces in a **right-handed** basis, and a road frame's is not: `n` is the LEFT normal `(t.z, -t.x)` while `tangent × up = (-t.z, t.x)`, the exact opposite. `pushPost` therefore walks its corner ring **+across → −across**, the reverse of `hexa`, and that cancels the handedness. Get it wrong and nothing complains: positions are right, the mesh still covers pixels, and every pin passes. The test is **signed volume via the divergence theorem — positive is outward** (the first version measured −389 against a `hexa` control of +8). In-render, the tell is that a correctly wound solid lights directionally; inside-out geometry reads flat from every side.

**Water (`physics.js` `_stepWater` + `water.js` + `env.setWater`).** Opt-in via `world.water`; null for pre-water scenarios so the path is skipped entirely. The ground stays a single 220 m slab UNLESS a basin exists, in which case it becomes four land slabs around the hole plus a bed collider — and **those land slabs are 20 m thick on purpose**: at 2 m a submerged car slid out from under them and fell forever. Submerged bodies get saturating buoyancy (0.62 g, deliberately under neutral so wrecks settle instead of bobbing) plus linear and angular drag. Determinism rules match the driver controller: arithmetic only, and thresholds compare **squared** magnitudes — never `Math.hypot`, whose last bits are not guaranteed across engines. Everything in `water.js` is render-side and cleared in `destroyRound`.
- **The visual ground never had the hole the collider did.** The sim carves a rectangular basin out of the ground collider; nothing carved it out of the ground *mesh*, so the flat y=0 surface ran straight over the channel and the water at y=-0.8 was drawn entirely behind it. Measured on seed `w5`: hiding the terrain put the surface on 72 % of the frame, leaving it visible put it on **exactly 0**. The basin had been invisible in every generated causeway — the `water` pin never caught it because a pin hashes bodies, not pixels. `punchBasin()` discards the rect **per fragment** on both the terrain and the ground disc; a geometric carve would land a ragged boundary on a polar mesh and then need the walls outset to hide it, which moves the shoreline off where the physics edge actually is.
- **The banks stay vertical** because the land slabs are vertical walls — a sloped visual bank promises a beach the sim will not honour. Riprap is what real causeways use, so an armoured edge is the correct answer that is also the legal one. Silt displaces **downward only**: the bed collider's top face is exactly `bed`, so upward displacement buries the settled wreck the player is watching.
- **Surface is `MeshStandardMaterial` + `onBeforeCompile`, not a raw `ShaderMaterial`** — it has to share the lighting, ACES, fog and shadow chain with everything else, and a raw shader means re-implementing all four and watching them drift the moment weather touches exposure. The wave sum lands in `<beginnormal_vertex>`, NOT `<begin_vertex>`: three builds `vNormal` in `<normal_vertex>`, which runs first, so a normal written beside the displacement is a frame late and the surface lights flat. Wind drives amplitude and foam (clear 0.063 → storm 0.225) so a storm churns instead of raining on a mirror.
- **`world.water.v2`** is an opt-in on top of the opt-in: buoyancy and drag reach dynamic props and torn-off wheels rather than only `this.cars`, in fixed order (props → each rec's `dyn` → debris). Absent on every shipping scenario, so `water` and `carnage` did not move. Buoyancy is a mass-independent acceleration, so `buoy` is a per-CLASS choice about whether that class floats (props 0.78 sink, tyres 1.05 float), not a density. Non-car entries fire **`onObjSplash`, never `onSplash`** — recorder.js resolves that one through `sim.cars.indexOf(car)`, so handing it a prop indexes `per[-1]` and throws inside the pre-sim the whole betting game settles against.

**Scenery (`scenery.js`) — 184 kinds in 8 categories.** `{ id, label, icon, cat, build(r, M) → {g, bodies} }`, same contract as `vehicles.js`: same kind+seed ⇒ identical model AND identical collider recipe, zero `Math.random()`. Categories: Nature · Suburbia · Street & City · Signs & Traffic · Water & Coast · Mountain & Alpine · Rural & Farm · Industrial. **Adding entries is safe by construction** — the generators pick kinds from hardcoded dressing lists, so new models are inert until something names them, while the registry auto-propagates to showroom, counts, crosshair labels, the smoke test and the contact sheet. Heavy things use `fixedBody`, knockable things `dynGround` (which re-roots visuals so the body origin sits at H/2). Traps worth knowing:
- **`cyl(mat, {axis: 'x'|'z'})` IS a rotation** — it sets `rotation.z = -π/2` (or `rotation.x = π/2`). Writing `.rotation.set(…)` or assigning that same component afterwards silently *undoes* it: a felled tree stood back up and beached driftwood turned into fenceposts, and neither failed the smoke test. Only `.rotation.y` is safe to assign on an axis-built cylinder; for anything more, wrap it in a group. The roof idiom (`rotation.x = π/2; rotation.z = π/4` after `axis:'z'`) overrides deliberately and is fine.
- **Nothing batches, so every mesh is a draw call.** A three-high container stack came out at 177 meshes — 3.5× the heaviest building — purely from corrugation ribs. Distance-read objects get an explicit LOD flag (`containerBody(…, near)`); the library's ceiling is ~68 meshes (`transmission_tower`, an irreducible lattice).
- **You cannot subtract one solid from another here.** An open hull, a bay, a trough has to be built as a *shell* — floor plus canted sides plus ends. Nesting a dark slab inside a light one to fake an interior just hides the dark one.
- **Silhouette beats surface.** Crags that only dent a face leave a box reading as a box; they have to break the outline, which means pushing past the top edge and off both ends.

**Props (`props.js`) & environments (`env.js`).** Props return `{group, bodies}` with *explicit* collider recipes, never parsed from geometry. Dynamic prop nodes re-parent to `sim.root` and sync from their bodies. Environments are strictly visual: preset palettes + deterministic decoration over the shared ground. **`ENVS` is 10** — `proving · salt · night · grid · city · dawn · dusk · alpine · coastal · desert`; `city` exists because `topoIntersection` and `topoRoundabout` have always asked for it (`director.js:267, 478`) and always silently got `proving`, since `apply()` falls back on an unknown id. The fix was to add the missing preset, never to edit the director — that keeps every generated scenario byte-identical. **An env id a topology names but `ENVS` lacks fails silently**, so adding topologies means checking both ends.
**Adding a preset is safe by construction and is a 6-table edit.** Nothing ever *picks* from `ENVS` — the director and worldgen name every id as a literal — so new entries cannot shift a generated scene and no pin moves (same argument as the scenery registry). They are reachable immediately because Settings lists `ENVS`. All six tables must be fed or something fails quietly: `PRESETS` (lights/ground) · `SKIES` (dome/sun/clouds) · `TERRAIN_FOR_ENV` (terrain.js) · `TERRAIN_VALUE` (env.js — baked colours no light can reach, so a low-sun preset must go well under 1 or its hills read as midday) · `WATER_PRESETS` (water.js) · `WEIGHTS` (weather.js). `tests/weather.mjs` is the backstop for the last one and asserts `ENVS`/`WEATHER_ENVS` agree **in both directions**.
- **Presets are PLACES and TIMES, not conditions.** The G6 plan originally listed `overcast` and `storm` here; after weather landed that says the same thing twice — a storm is expressible on every preset and `WEIGHTS` decides where it fits, and a storm *place* has no coherent answer for what it looks like in clear weather. `desert` replaced them, and `dawn`/`dusk` carry the time-of-day axis the 1B sun rig can finally express (`el ≈ 0.12` gives genuinely raking shadows). Warm key against cool hemi is what sells low sun; warm *ambient* just reads as an orange filter.
- 1F also put three authored-but-unreachable landscapes into play: before it, `TERRAIN_FOR_ENV` only ever resolved to `rolling` or `flats`, so `alpine`, `mesa`, `dunes` and `coastal` existed with no env that asked for them. Fog near/far live in `env.state` and are scaled by `env.setFogScale(fk)` — don't set `scene.fog` directly. **Skybox (reworked in 1B)**: a camera-following **shader dome** + sun/moon + deterministic stars, parallax cloud blobs and a scrolling cloud deck, all `fog:false, toneMapped:false`; `env.syncSky(camera.position, dt)` runs every frame.
- **Structure is per-PRESET, weather is uniforms.** `makeSky(id)` is called only from `rebuildSky()`, which only `apply()` calls; `syncSkyTint()` handles weather with uniform writes and zero allocation. `makeSky` used to run from `applyWeather`, so **every weather change disposed the dome, its canvas textures, the star field and every cloud blob and rebuilt the lot**. Verified: dome/material/geometry UUIDs and the program count are unchanged across six weather changes, and 30 preset×weather cycles leak zero geometries or textures.
- **The gradient is injected into `MeshBasicMaterial`, not a raw `ShaderMaterial`** — same call water.js makes. `toneMapped:false` plus the material's own colorspace chunk *is* the linear→sRGB-without-tonemapping path the old CanvasTexture got for free; hand-rolling it is how authored hexes drift from what ACES does to everything else.
- **The CPU still computes top/mid/hor with the same `mixHex` expression** and passes them in, because `horizonNow()` also feeds the terrain's baked haze, `scene.background` and the fog tint — those four must agree or the landscape meets a sky it does not match. The shader only adds what a vertical ramp cannot express: **azimuthal sun glow** (ledger #2) and a horizon haze band. Measured on salt: luminance 246 at the sun's azimuth falling to a flat 189 opposite, where a ramp gives 189 everywhere.
- **The cloud deck projects onto a flat layer** (`d.xz / max(d.y, 0.12)`), so it foreshortens toward the horizon like a real cloud layer rather than sitting on the dome like wallpaper. Higher cover *lowers the fbm threshold* so the deck closes up rather than fading in — fading in reads as dirty glass, closing up reads as weather. Hidden below cover 0.06, so a clear scene pays nothing. 0.06 ms; the whole sky is 0.18 ms.
- Stars carry per-star size, brightness and colour off one seeded stream, brightness **cubed** so the distribution is faint-weighted (a handful of bright ones, not an even wash of mid-grey). Sizes 1.1–3.2 px against the old 260 identical 2.2s.
**The preset/weather split** is what makes per-scene variation expressible at all: `apply(id)` rebuilds STRUCTURE (sky mesh, decoration, terrain, ground texture, light *colours*) and early-returns on an unchanged id; `applyWeather(w)` moves intensities, fog distances, haze and cloud cover, and **never** early-returns. The old single `apply` did both and bailed when the id matched, so "same preset, different weather every scene" could not be said. `apply` ends by calling `applyWeather` so a preset change re-lands the current weather rather than dropping it.

**Terrain (`terrain.js`) — and it IS the ground.** The landscape around *and under* the play area, owned by env (`env.setTerrain(spec)` / `env.terrainField`), driven from `world.terrain` or a bare seed. It reaches r=0, so while it is up the old flat `CircleGeometry` disc **hides rather than being deleted** — the showroom and crash mode still want a plain apron, and a hidden mesh costs nothing. Inside `playR` the height mask is exactly 0, so the play area is the same flat y=0 plane the disc was, receiving shadows the same way; what changed is that the surface colour is a radial ramp **in world units** instead of a stretched CanvasTexture (which also retires the bug where `setGroundRadius` rebuilt the geometry but never rescaled the texture, so one gradient stretched identically over 90 m or 167 m), and the world no longer ends at a visible rim. **Ring 0 collapses to the origin**, so its second triangle per quad is zero-area — emit it and `computeVertexNormals` hands back a zero-length normal and a black speck at the scene centre; ring 0 is a triangle fan and the vertex buffer is trimmed to what was actually written. **Opt-in with null exactly like `world.water`** — no spec, no mesh, no cost, and that is why every sim pin survived it. One polar mesh, never a near disc plus a far ring: `rings × spokes` with radii packed as `u^1.7` so detail lands just outside `playR` where the eye is. Displacement is masked by `smoothstep(playR, playR*1.35, r)`, so it is *exactly* 0 on the drivable area — terrain can never intrude on a lane and there is no seam to hide. Height is integer-hash value-noise fbm with an optional ridge fold (`1-|2n-1|`, squared) and a domain warp, because unwarped ridgelines run along the noise grid and read as corduroy. Vertex colours (rock/grass/scree/snow by height and slope), non-indexed + `computeVertexNormals()` for the faceted read, one `MeshStandardMaterial`, one draw call. `makeHeightField(spec)` is exported separately so vegetation scatter and anything else that must sit ON the landscape queries the same surface the mesh was built from.
**Drivable terrain (P2/2C)** is opt-in on `world.terrain.drivable`, the same null-opt-in as `world.water` — absent, not a line runs and the ground is the flat slab it always was, which is why every pin survived. Until it, terrain was **visual only**: the collider is a flat 220 m slab whatever the landscape does, so a car past `playR` drives straight through the hillside (measured over 60 recorded scenes: 4 put a car inside terrain, worst burial 4.6 m — rare, since it needs a car to escape the play area first). It samples the same field the mesh uses, so the two cannot disagree: measured worst |Δ| **0.004 m** over the full ±154 m extent, and cars settle within 0.04 m of the visible ground.
- **Rapier's layout is row-major `i*(n+1)+j`, i along x and j along z.** Measured with an asymmetric ramp, not assumed — the column-major reading is exactly the TRANSPOSE, which renders as entirely plausible terrain that is wrong at every point. Probe with asymmetric pairs like (80,20) vs (20,80); a symmetric probe or a diagonal cannot see it.
- **Extent is sized off the ARENA, not off `rampTo`.** `rampTo` reaches only 1.43 × playR and a relief topology deliberately keeps playR small so the hills are near the action, so the grid stopped inside the landscape and everything past it silently fell through to the slab (the tell: probes reading exactly 0.00). Resolution is a **cell size**, not a grid count, so widening the extent cannot quietly coarsen the surface under the cars.
- **The grid is memoised on the spec.** Sampling is ~1.6 µs a call, so a 194² grid cost 61 ms — on its own more than the ~30 ms "perfect reset", and a reset happens on every freeze-scrub capture. Cache hit is 7.3 ms and verified **bit-identical** to a cold build over 200 ticks; the array is a pure function of the key, so that holds by construction.
- Rest detection survives slopes, which was the plan's stated worry: cars dropped on 3°/6°/25° slopes reached rest at ticks 95/94/750 (the 25° case slid 100 m first, well inside the 2400-tick cap).

Three things that cost real time:
- **Winding is the same trap as roads.js.** Spoke index runs with +angle and ring index with +radius, so the natural `(00,01,11)` order yields a −y normal and the entire landscape is backface-culled from above — present, lit, invisible.
- **Amplitudes are set against this world's scale, not the real one.** Play area ~90–170 m against a 430 m dome means hills only read between roughly 30 m and 120 m; at 20 m a ridge 400 m out subtends nothing and the ring renders as a flat wash. The far band gets a ×3.6 boost specifically to break the skyline.
- **A polar grid stripes.** Every vertex on a ring sharing one radius aligns height contours with the rings, and flat shading turns that into concentric bands down every hillside. Each vertex is jittered within its own cell (ring 0 pinned, since it meets the ground disc) — topology unchanged, lattice broken.

**Weather (`weather.js`).** Per-scene weather. The RENDERING half is strictly render-side like fx.js: it reads nothing from the sim and writes nothing back. `rollWeather(seed, envId)` and `gripFor(wx)` are the other half and are **pure** — plain data, no THREE, no side effects — which is what lets director.js call them. `rollWeather` draws from its OWN rng stream (`'wx:'+seed`), so the same seed always shows the same sky and no existing stream shifts by a draw. 11 kinds, weighted **per environment** because weather has to fit the place (a desert blizzard reads as a bug, and `grid` stays clear so the diagnostic preset stays legible). `destroyRound` clears the sky and the particles so the rain does not follow you into the menu.
- **The SCENE carries its weather** (P2/2D): `generateScene` rolls it and puts the descriptor on `world.weather`, and main.js reads `sc.world.weather` rather than rolling its own. It has to be this way once grip reaches the tyres — the recorder that priced the odds and the round the player watches must not be able to read different weather. Verified behaviour-preserving over 400 scenes: the descriptor is byte-identical to what main.js used to roll, because every topology names an env `WEIGHTS` has a table for. The roll survives in main.js only as the fallback for scenarios that never went through the director (crash mode, the pins). It also retires a latent trap — main.js rolled against `env.current`, the LIVE preset, which is a Settings chip the player can change.
- Until P2/2E only **4 of the 10 env presets were ever dealt** (`proving`/`city`/`salt`/`grid`); 1F's `dawn`/`dusk`/`alpine`/`coastal`/`desert` were reachable only from the Settings chip, so `snow` and half the weather table could not occur in a real round. **2E's twelve topologies fixed that** — they name dawn (tunnelmouth/cloverleaf), dusk (boulevard), alpine (mountainpass/forestroad), desert (canyon), coastal (coastalcliff/riverside/harbourramp) and night (industrialyard), so every preset and every weather kind is now dealable.
- **The lighting is the feature; the particles are garnish.** An overcast scene reads as overcast with no rain drawn at all. `key/hemi/fill/exposure/envI` are MULTIPLIERS over whatever the preset authored, so weather composes with a place instead of replacing it.
- **`envI` (`scene.environmentIntensity`) is the biggest lever and the easiest to miss.** The PMREM RoomEnvironment is a constant IBL flood that no light intensity touches; leave it at 1 and a storm on a pale preset is a bright afternoon that merely lost its shadows. Wired through `ctx.setEnvIntensity`, next to `ctx.setExposure` — and `BASE_EXPOSURE` (1.18) stays the look-tuning baseline that weather multiplies, so a clear scene is bit-for-bit what the game was tuned against.
- **`hemi` goes DOWN in bad weather, not up.** The instinct is to raise it — overcast really is a sky-wide source, the shadows really do go away — and the first table went to 1.38. On a white salt flat that produced a storm you could read a newspaper by: albedo wins, and killing the key alone cannot darken a surface the ambient is still flooding. It rises only where the ground genuinely bounces (snow, mist). The key:hemi *ratio* still climbs, which is what actually sells flat weather light.
- **Haze is a colour, not a grey.** Each kind carries `haze {hex, amt}` — dust is tan, mist is *brighter* than what it hides, a storm is near-charcoal — and the sky gradient, `scene.background`, the fog tint and the terrain's baked horizon must all derive from the same `hazeTarget(preset, haze)` expression or the landscape meets a sky it does not match. Amount is `max(cloud, fog×0.95)`: cloud alone under-reads dust, which hazes hard on a half-clear sky. Cloud and haze are separate knobs on purpose — cloud hides things *in* the sky (sun, stars, blob count), haze recolours the air, and driving blob count off haze grows clouds out of a dust storm.
- **Wet roads (`applyWetness`) do more work than the rain.** A downpour falling on bright dry asphalt reads as a bug and no particle density fixes it. Roads only — materials come from a per-build `matFactory` cache so nothing leaks into the next round, and terrain is left alone (grass and rock do not gloss). Scalar-multiplying a linear-space `m.color` is an exposure change, not an HSL round-trip, so it is exempt from the sRGB rule.
- **Precipitation is two pooled systems with zero per-frame CPU** — fall, wind drift and wrap all happen in the vertex shader against a 44×30×44 box that follows the camera, so changing the amount is a draw-range write. The box is deliberately small: density is count/volume, and a box big enough to "cover the scene" spread an affordable drop count into six visible streaks. Rain is `LineSegments` because `gl_PointSize` is square and cannot express a velocity-aligned streak. **Flakes need ~2.5× the drop count for the same read** (a drop crosses many pixels, a flake is a dot; at rain counts snow was ~20 specks and read as a dirty lens). Whole system costs **one draw call**.
- **Flakes size in metres through the projection** (`aSize × uPix / -mv.z`, `uPix = h / 2tan(fov/2)`, refreshed in `resize`) — sizing in raw pixels makes near flakes 200 px ping-pong balls. Correct sizing alone still leaves the one flake that drifts within a metre of the lens rendering as a fat soft ball, so there is also a `smoothstep(0.7, 3.5, z)` near-fade; no real lens resolves what is inside its near focus.
- Weather is **frozen while the round is** (`update(dt, camera, paused)`): the sim's clock is stopped so hanging rain is the correct fiction, and it lets render-on-demand sleep through the freeze — the longest UI phase, and exactly when the player wants a steady frame to read.

**Weather grip (P2/2D) — where weather stops being a look.** Opt-in on `world.weather.grip`, the same null-opt-in as `world.water` and `world.terrain.drivable`: absent, `wxGrip` is exactly 1, and multiplying an IEEE-754 float by exactly 1.0 is bit-exact, so every scenario builds the tyres and brakes it always did. All 11 prior pins held; `wxgrip` is the 12th and freezes the feature itself. **P2/2E turned it on**: mountainpass/canyon/forestroad/coastalcliff/riverside set `wxGrip`, so `gripFor` now reaches the tyres in real rounds — a wet alpine pass or coastal road genuinely lengthens a stop.
- **It is TWO code paths, and only the second one made it work.** The obvious one is `setWheelFrictionSlip`, scaled at build and again in the bent-wheel penalty (which re-derives slip from scratch, so without it a wheel gets its wet-road penalty *refunded* the moment it bends). That alone was not enough: `setWheelBrake` is fed a fixed `mass × brakeK` force that sits well under the dry friction circle, so **peak deceleration measured a flat ~13 m/s² from grip 1.0 all the way down to 0.7**, only starting to bind near 0.35. Wet roads had dry stopping distances and grip perturbed nothing but lateral scrub.
- **The failure that mattered was directional, not mechanical.** With friction slip alone the mechanism was correctly wired and provably safe, and it still failed the product bar: settling every market dry vs wet, **0.82 flipped MORE markets than 0.70** (4.2 % / 4.5 % / 4.0 %, 47 % / 44 % / 37 % of scenes). Grip was re-rolling the dice rather than pointing anywhere, which is exactly the "unfair randomness" the product rules forbid — the player must be able to see rain and *reason* about it. Scaling brake authority by grip too made it monotone: **5.1 % / 5.6 % / 5.7 % of markets and 61 % / 65 % / 69 % of scenes** at grip 0.82 / 0.76 / 0.70. Same wiring, same safety, and now a relationship a player can learn.
- **A hash moving is not evidence.** 95.6 % of scenes hashed differently under grip, which sounds decisive and proves nothing — one ULP on one body does it. The measurement that settled the design was `settleMarket` over every market the game would actually offer.
- **Grip is per-KIND, not derived from `wetness`.** `wet` describes how GLOSSY the road looks and the two genuinely disagree: snow barely wets asphalt (`wet` 0.20) and is the slipperiest surface in the table, dust is bone dry and a layer of grit really does cost grip. Deriving grip from wetness would have made snow a 0.95 and dust a no-op — backwards on both counts.
- **Clamped to 0.35–1**, because below roughly 0.35 the driver cannot hold a lane at cruise and cars understeer off the road *before* the incident, which breaks the tick-600 rule rather than making a scene interesting. And the guard is **`Number.isFinite`, never `typeof === 'number'`**: NaN passes a typeof test, `clamp` propagates it (Math.min/max of NaN is NaN), and it reaches Rapier's wasm as a bare `unreachable` panic naming nothing. Measured, not hypothetical.
- **CALIB has no weather axis, and 2E has to decide what to do about that.** `CALIB` is keyed kind → template → bucket, so once a topology enables grip its wet and dry scenes are priced from one blended rate — mispricing both, by an amount of the same order as the house margin. Do not add the axis on a hunch: CLAUDE.md's own rule is that a correction needs a calibration sample bigger than the gate validating it. Measure the wet/dry rate gap at regeneration time and only split the bucket if it clears the estimator's noise.

**Vegetation (`vegetation.js`).** Instanced scatter over the terrain ring, render-side like fx/weather: it reads `env.terrainField` and writes nothing. No colliders ever, and masked to `r > playR * 1.06`, so it cannot touch a lane or a hash. `veg.build(field, seed, {density, value})` from `startScene`, `veg.clear()` from `destroyRound`. Currently **7 draw calls for 1250 plants**, ~2 ms and ~190 k triangles on desktop (420 plants on small screens).
- **One InstancedMesh per KIND, not per (kind, material).** `matFactory` dedupes by parameter key, which is the wrong granularity here — every `jitterColor` call mints a new hex and therefore a new material, so `scree` arrives with 37 materials for 41 meshes. Instead each prototype is merged into ONE buffer with the per-mesh material colour baked into a **vertex-colour attribute** under a single white `vertexColors` material, and per-plant variety comes from `instanceColor` multiplying on top. `m.color` is already in the linear working space and vertex colours are consumed in that space, so this copies numbers rather than converting them.
- **Scatter is near-biased (`u^1.9`), not area-uniform.** Area-uniform is the textbook answer and it was visibly wrong: area grows with r², so ~80 % of plants landed in the outer third and from a dashcam the roadside was bare. Same packing trick terrain.js uses on its rings, same reason. Species also carry a `reach` — a blade of grass 250 m out is a subpixel that still costs a matrix.
- **Sway is `h² · amp` in the vertex shader**, injected after `#include <begin_vertex>` where `transformed` is still plant-local; phase comes from `instanceMatrix[3].xz` so neighbours are never in step. The quadratic *is* the trunk mask — no per-species tagging needed, and it keeps a 0.6 m rock under a centimetre for free. Amplitude was 4× too small at first and the bug was invisible: correct, deterministic, 3 cm, and indistinguishable from a still frame. **Verify sway with a pixel diff across ~0.75 s, not by reading uniforms.**
- Plants carry baked colours that no light can reach, so `build` takes `env.terrainValue` — the same expression the landscape was baked at. Without it a night scene or a downpour leaves vivid daylight-green trees on grey hills.
- Frozen during the freeze and under reduced motion, same reasoning as weather.

**World generators (`worldgen.js`).** `generateWorld(preset, seed, {maxProps, maxRoads})` (suburb / city / highway) emits plain scenario content, so generated worlds are ordinary editable objects. One rng stream per `'w:'+preset+':'+seed`; placement walks the spline with lateral zones chosen so nothing overlaps by construction. Props push identity-first (anchors → lots → filler) and trim to `maxProps`, so smaller budgets drop filler, never identity — but note the same seed gives a *smaller subset* on mobile, and that trim order is part of each generator's design. Gotcha: the city loop's left normal points OUT of the block, so outside-ring props use positive lateral offsets and the plaza uses centroid math.

**Effects (`fx.js`).** Strictly render-side; reads sim state and hooks, never writes back (fx has its own `makeRng('fx')`). Pooled zero-alloc systems: four shader point clouds (sparks / normal-blend puffs / fire / water spray), instanced glass shards and paint chips, a ground-decal ring buffer (skids, scrape grinds, fluid spots — deterministic y micro-offsets kill z-fighting), an impact flash PointLight whose colour is set per trigger, and camera shake applied around `renderer.render` only then undone. Struck props start *standing* emitters keyed on `props[i].spec.kind` — geysers (hydrant), arcing sparks (poles/lamps/signals) — anchored to the prop's SPEC position, not its group, because these props are dynamic and the body tumbles away while the sheared main stays planted. Fire spread is render-only ignition in an fx-local map (`frontDmg` is sim state and fx must never write it) and only a shell already past `damage > 9` catches. `initFX(scene, {small})` halves pools on small screens; `attach(sim)` wires the hooks and re-keys `perCar`; `update(dt, camera)` returns busy so render-on-demand still sleeps when idle.
**Audio (`Sfx`)** is procedural and zero-asset. The engine bed is a second sustained voice alongside the scrape loop, which works only because it occupies a different band: two detuned saws at 55–190 Hz under a 340 Hz lowpass vs the scrape's 1050 Hz bandpass. UI sounds go through `_ping`, which deliberately does NOT consume a crash one-shot slot — UI feedback must not be dropped because six cars collide in one frame. Volume lives on the master gain; `mute()` restores to the saved volume. The context needs a user gesture, so `?crash=N` boots register a one-time pointerdown unlock.

**Incident engine (`director.js`).** `generateScene(seed, d)` deals a complete deterministic round from separately derived rng streams per facet (`'scn:'+seed+':topo'/'incident'/'cast'/'dress'`). **22 topologies** (bespoke signalized intersection + suburb/city/highway from `generateWorld` + causeway · switchback · schoolzone · tramcrossing · parkinglot · roundabout, **plus the 12 P2/2E places**: boulevard · tunnelmouth · industrialyard · tjunction · overpass · cloverleaf · forestroad · mountainpass · canyon · coastalcliff · riverside · harbourramp); **28 incident templates** (the original 20 + the 8 P2/2F ones: rockslide · fallentree · overheight · barrierdrop · fogbank · wideload · lowgrip · flooddip), each returning cars with per-tick `drive` specs (polyline + cruise speed + sparse `cmds`) and optional props.
**The 2F eight** are each built on a proven-safe pattern (debris obstacle-swerve / chain same-speed convoy / blowout carve) so the nothing-before-600 rule holds by construction, and several give the 2E places incidents that fit — a rockslide on a pass, a fallen tree in the forest, a flooded dip by the river, an overheight trailer at the tunnel. `lowgrip` and `flooddip` are the first templates authored to READ as weather-grip bets (`lowgrip` deals only on wxGrip topos — the wet road, not luck, decides the stop). Each has a `specialFor` case in markets.js (**18 of 28 now**, was 8 of 20). CALIB has no rows for the new templates, so `calib()` falls back to the kind default — markets still generate and settle, and the close-gate CALIB regeneration prices them properly. The plan listed ten; `convoysplit` and `towstrap` were built and cut as redundant `chain` variants that also fought the lane model (queue overrun; 82% eventless). Adding templates did NOT move the director pin: none of the eight deal on `intersection`, so pin-1's (intersection/police) template pick is unchanged.
**The 2E twelve are the callers 2B/2C/2D were built for** — each of those shipped inert with no topology turning it on. mountainpass/canyon/forestroad set `world.terrain.drivable` (2C) + gravel verge / guardrail style bits (2B); mountain/canyon/forest/coastal/riverside set `wxGrip` (2D). They also deal the five 1F presets (dawn/dusk/alpine/coastal/desert) + night that only the Settings chip could reach, so snow and the cold half of the weather table finally occur in a real round. Adding names to the pick array moved NO pin: pin-1's rng still lands on `intersection` (floor(u·N)=0 for both N=10 and N=22) and that path is byte-identical, so the sweep — not a pin — is what certifies the new places (360/360 clean at PER_D 120). Two traps the batch cost: a signalized T-junction is impossible with this lane model (the stem's inbound lane ends in the junction centre with no opposite arm to exit through, and a junction-radius turn is tighter than the driver's ~22 m floor — so `tjunction` is a give-way SPUR with no through traffic, the incident on the main road); and curvy/relief topos (mountainpass/canyon/forestroad/coastalcliff) bar heavies via `NO_HEAVY`, because a long semi off-tracks across the centreline on a sustained gentle bend — the `bendy` v²·κ guard measures the tip, not the length-driven crab, and waved it through into a pre-incident head-on. Topologies that meet at grade emit `junctions` (intersection: bars/walks/box/arrows; tramcrossing: keep-clear only; roundabout: four 2-arm **aprons**, one per stub mouth, which is where the real seam was — the old patch prop sat on the grass island and hid nothing). **Everything goes wrong at exactly tick 600.** Hard-won invariants:
- **Nothing may collide before tick 600.** A conflict scrub staggers crossing arrivals, then a spawn-overlap net runs BOTH before and after the scrub, since the scrub moves cars.
- Head-on templates (`needsOpp`) only deal where opposite-lane geometry exists, and the victim's arrival window accounts for COMBINED closing speed (windows sit at ~T+70–120).
- Swerve cmds that must leave the lane need `off: true` or pure pursuit cancels the bias. Heavies are barred from fast bendy plans (`v²·κ > 2.4`).
- Dressing is scrubbed off lane corridors, out of big-prop footprints (a tree inside a house collider gets ejected and falls across the road), and away from car spawns.
- **`sortLanes` quantises its length key to cm and tie-breaks on build order.** The two directions of a symmetric road are equal to within float noise, and a raw `b.len - a.len` let `sqrt`'s last bits decide — node and the browser sorted a causeway's lanes oppositely and **generated mirrored scenes from one seed**, which breaks the core promise that the pre-sim the bets settle against is the scene the player watched.
- **`place()` may drop below `MIN_V` to hit its arrival tick exactly.** Lanes are pre-filtered for a 600-tick run, but templates aiming past T=0 need more road; the old floor clamped `s0` to the lane start so the car arrived EARLY and rear-ended a still-parked victim at tick 599.
- **The near-miss shift is capped by the room the victim can give up** and still reach its anchor after T=0. Shifting it up-lane also makes it arrive sooner; unbounded, that silently undid the arrival budget in both directions. The spawn-overlap net cannot catch this class — the cars never overlap at spawn, they converge early.

**Traffic signals (`signals.js`, P2/2I).** A program is plain data and `signalAt(prog, arm, tick)` is a **pure function of the tick** — no rng at eval time, no state between ticks, integer arithmetic. It has to be: drivers obey it, so it settles money, and a freeze-scrub that jumps backwards must get the lights the forward run had. Arms are keyed by the string lanes already carry as `road` (`'ew'`/`'ns'`). Programs live on `junctions[i].signal`; `physics` evaluates, `main.js` mirrors the aspect onto the lamps (tagged `userData.sigLamp`, `noMerge` — a merged lamp cannot be lit independently of the pole it fused with).
- **An unknown arm returns GREEN, and getting that wrong parks every car on an unsignalled approach.** The first version decided "is this arm governed?" *inside* the stage walk, which returns early — so an arm appearing only in a later stage was reported red on the strength of stages it was never in. `governed` must be settled before the walk.
- **Intergreen must cover junction clearance.** Amber + all-red has to exceed the time a car that entered on green takes to cross (~19 m at ~12 m/s ≈ 1.6 s). At the original all-red of 20 ticks the intergreen was 1.1 s and cars legitimately mid-junction were T-boned by traffic just waved through. Now 45 + 80 = 125 ticks ≈ 2.1 s.
- **Phase by SEARCHING the offset, not by pinning one (arm, tick).** An intersection incident almost always involves actors from both arms — that is what makes it one — so `phaseFor` alone left 74 % of junctions unable to bind. A two-stage cycle can be green for `ew` at tick 250 and `ns` at 590; it just needs the right offset, and the space is one period of cheap integer tests. Binding went 26 % → 76 %.
- **Obedience is all-or-nothing per scene.** Essential cars never carry stop lines (their run-up is budgeted at free-flow cruise), so the signal is arranged *around* them. If no offset covers every actor's crossing, ambient traffic ignores the signal entirely and the lights become cosmetic for that scene. **Half-obeying is what breaks the tick-600 rule**: with only the cross street obeying, an ambient car on the actor's arm sailed through its own red while cross traffic legitimately moved on green — 4 pre-incident hits in a 75-scene sweep, all on the intersection.
- **Stop lines stop binding at `INCIDENT_TICK + 150`.** Rest is measured by displacement and a signal cycles ~3.5× over the 2400-tick cap, so obedient traffic is perpetually restarted by a fresh green and the scene never settles (ran-to-cap 6.7 % → 10.7 %, breaking the sweep budget).
- The driver holds at the bar via the same anticipatory `v² = 2as` shape as end-of-path, **plus a firm hold** — the limit alone decays asymptotically and leaves a car creeping at 0.6 m/s, which over a 200-tick red walks it into the junction. Amber is a **dilemma zone, not a second red**: a car too close to stop comfortably must go, or you manufacture the rear-end the whole file exists to prevent. The done-latch lives on the DRIVER, never on `drive.stops` — the scenario is shared between the recorder sim and the live one.
- Cast size is **4–10** (was 3–8), with up to four cars deliberately queued on whichever arm is red at the freeze. Queue cars all aim at the same bar at staggered arrival ticks; stagger by **distance, not a flat tick count**, because the spawn-overlap net culls non-essential pairs closer than 8 m and a fixed 30-tick gap gave 6 m at 12 m/s — it quietly ate every queue down to two cars.

**Driver controller (in `physics.js`).** Cars with `spec.drive` get closed-loop pure pursuit: arithmetic-only (`+,-,*,/,sqrt,min,max,abs` — no transcendentals, so bit-deterministic across engines), anticipatory `v² = 2as` braking, `end: 'stop'|'coast'`, sparse cmd timeline, post-crash shock (damage > 6 → brake and hold), and **ACC car-following for ambient cars**: path-probe detection (goal + far + midpoint — a straight-ahead cone misses leaders around bends), a same-direction gate (dot > 0.2, since reacting to crossing traffic shifts junction timing and breaks the conflict scrub), capping to the leader's TRUE velocity projected on own heading, full attention only before the incident tick. Driven cars get real brake authority (`brakeK` 0.055 vs the legacy 0.02, which keeps old hashes frozen), **scaled by `this.wxGrip`** since P2/2D — the tyre model alone does not bind in the braking range, so wet roads only lengthen a stop because the demanded force falls with grip. `sim.stopAt` freezes on an exact tick; `colToObj` maps collider handles → `{kind}` for impact attribution.

**Pre-sim recorder (`recorder.js`).** `recordScene(...)` runs the scenario headlessly to rest and returns `{events, summary, restTick, ticks, hash, tracks}` — the log the whole betting game settles against. **Rest is measured by DISPLACEMENT, never instantaneous velocity**: three steady states read as motion forever while a body sits still — raycast suspension holds a constant +lv.y ≈ 0.73, buoyancy holds another, and a stuck car under driver control yaws in place — and the old `vmax` test converged on none of them, running 48 % of scenes to the cap. The threshold is derived, not tuned: 0.6 m of drift across a 75-tick window caps a body under 0.5 m/s, so no two bodies below it can close hard enough to log a contact (`TOUCH_DV` is 1.4). It must be that generous because `end:'coast'` cars decay asymptotically. Deliberately exempt from the gate: disturbed **props** (markets latch on first move, and a knocked cone rolls forever) and **sunk** cars. Events are fixed-order read-only scans inside the step loop; a new event kind MUST get an `EV_CODE` or it hashes as `undefined`. The FNV event hash is the determinism fingerprint (bit-identical node↔browser).

**Betting core (`markets.js` + `economy.js`) — headless-first.** `generateMarkets(scene, {labelOf})` is **outcome-blind by construction** — it never imports recorder.js; inputs are the scenario plus the static `CALIB` table of empirical hit rates keyed kind → template → bucket. Every market carries its `cal` bucket tag so the harness measures exactly what the generator prices. Odds are decimal ×100 integers, `oddsFromP = (1 − marginFor(d)) / p` clamped ×1.10–×50; margin eases 6 % → 2 % at d ≥ 8. Offering rules learned from the Monte Carlo: markets with p > 0.85 clip at the odds floor and pay the BETTOR, markets with p < 1.5 % clip at the pricing floor and pay the house — neither is offered; the headline pair is exempt (the spec requires both sides) and gets a ×1.03 floor. `settleMarket(market, rec)` is the only outcome reader.
Two calibration rules that cost real debugging:
- **`tools/calibrate.mjs` MUST pass `{ all: true }`.** Offering is itself a function of CALIB, so measuring rates over only the offered set feeds a truncated sample back in as the full-population prior. That loop does not converge: regenerating once put the house edge at +14.6 %, splicing and regenerating again flipped it to −42.2 %. Calibrating over every candidate breaks it — the table then describes the world instead of the previous table's selection.
- **There is deliberately no difficulty axis on CALIB.** When the high bands ran under target, two corrections were built and both made the gate WORSE (d8–10 to −7.7 %, then −4.5 %, from −2.8 %). A marginal per-(kind, difficulty) rate double-counts scene composition CALIB already prices via role and template; an observed/expected ratio avoids that but could not be estimated better than ±0.06 from a 400-scene sample — larger than the effect. The real cause was a stale table measured against a build where cars fell through bridges. **Regenerate CALIB first**; reach for a difficulty correction only with a calibration sample bigger than the Monte Carlo validating it.

`economy.js`: injectable store (localStorage in app, memory in tests), profile v1, slip = singles + one parlay (integer stakes ≥ $1, total ≤ bankroll — the bankroll cannot go negative by construction), `placeSlip` deducts at bet time, `settleRound` pays ⌊stake·oddsH/100⌋ (stake included), pushes the ledger, advances the campaign, and handles ROCK BOTTOM. **Achievements** (`achievements.js`) evaluate only on money rounds — Exhibition seeds are re-runnable, so unlocking there would let a player farm every badge off one seed — and unlocks are append-only and idempotent. The **daily seed** is deliberately not a special case in the money path: a normal money round with a date-derived seed (no profile key, so everyone shares the scene), where the settled-seed ledger already enforces one attempt. Streak arithmetic parses the key at **midday** so a DST shift cannot move "yesterday" by a day.

**Game UI (`betui.js` + `povcam.js`).** `betui.js` owns every pixel of the betting layer and NO sim state — main.js drives the lifecycle and calls `openRound / setPhase / tickLive / settle / closeRound`; the only callbacks out are `onLock` and `onNext`, and audio is **injected** rather than imported so the module stays mountable with no AudioContext. **Live chips are a preview, not settlement**: `buildLiveIndex` precomputes per market the tick it becomes definitively won and definitively lost, so a chip flips the frame the player sees it; by construction the state at `restTick` equals `settleMarket`, and `settle()` overwrites every chip from the authoritative report regardless. Betting stays open through the freeze — **time restarting is the lock**. Bugs worth not reintroducing: re-rendering `#mklist` innerHTML on a tap resets scrollTop and throws the player to the top (tap path uses `syncMarketRows`); **there is deliberately no tap-anywhere-to-resume** — the freeze exists to study the scene and finish a slip, so a global pointerdown handler meant every tap aimed at a market, the crosshair or the camera restarted time instead; `#fzGo` and the space key are the only exits, and the freeze layer is a pass-through box (`pointer-events: none`, children re-enabled) so it never covers the scene or eats a click; a half-built parlay sits at one leg with a hint and blocks PLACE rather than silently dragging in another pick.
`povcam.js` builds the deterministic camera rig (`'pov:'+seed`): dashcams on flagged cars (the aggressor rarely gets one — riding it hands over the read), CCTV pole, chopper orbit, witness tripod; `povBudget(d)` prunes coverage and everything unlocks at `phase === 'done'`. Camera MOTION lives in povcam.js, the LENS treatment (grain, vignettes, letterbox) is CSS on `#povfx`.

**Freeze scrub — the filmstrip (`buildStrip`/`poseStrip`/`seekPreview` in main.js).** There is no rewind in a rigid-body world, so a backward seek used to `sim.reset()` and replay from t0 — correct, but ~30 ms of mesh rebuild plus up to 600 steps *per drag frame*, which crawled under the thumb. Entering the freeze now captures the last 10 s of **visual** state once (`buildStrip`) and a seek becomes a pure pose: **0.016 ms**, no physics, no rebuild, no allocation. Captured per tick is exactly what `syncVisuals` interpolates — chassis pose, steer, suspension travel, wheel spin, plus dynamic props; debris is not captured because nothing may collide before the incident tick. `poseStrip` writes `prev` alongside `cur` so the alpha lerp is a no-op while frozen and the first stepped frame after resume interpolates from what is actually on screen.
The payoff is structural, not just speed: **the sim is never rewound at all** — it stays parked on the incident tick, so `resumeRound` cannot hand the player a scene the odds were never priced on (verified: `hashState()` is bit-identical across every seek, and a resumed round reaches the recorder's exact `restTick`). Capture order still matters — `buildStrip` calls `reset()`, so fx must `detachSim` first, `targetMap` must be rebuilt from the fresh meshes, and the six hooks are nulled for the capture pass then restored by `attach` + `hookRoundCinematics`. **Since P2/2H every difficulty builds the strip** (the `d ≥ 8` freeze skip is gone), so `round.strip` always exists during a seek and the old re-sim fallback in `seekPreview` (reset + step up to 600 ticks) was deleted — `seekPreview` is now a pure `poseStrip`. `round.seekTo` still throttles it to one pose per frame.

**Quality tiers (1H).** `applyQuality(q, save)` owns DPR, `shadowMap.enabled`, shadow map size, vegetation density and the precipitation draw-range budget, and it **persists** in `profile.settings.quality` — it used to move only DPR and shadows, forced a full material recompile on every call, and reset to the screen-size guess on every boot, so a player who picked Low got High back each time. The recompile now happens only when `shadowMap.enabled` actually flips, which is the one change that alters the shader. Measured Low vs High on a 7-car scene: **4.2 ms vs 7.0 ms**. Note the boot call passes `save: false` and is guarded by `qualityLive`, because it runs at module scope *above* the `let round` / `let profile` declarations and reading either there is a temporal-dead-zone throw.

**Loading + motion.** `makeProgress` is ONE component reused three ways (boot preload · per-round pre-sim beat · scene loading) against shared `.pbar`/`.pfill` markup. **Reduced motion is a live setting, not a boot constant**: the OS media query seeds it and the in-game toggle can turn it ON, but the toggle never overrides an explicit system preference back to "full". It mirrors onto `body.reduced-motion` so CSS, the camera tween and the bankroll count-up honour one switch.

**`main.js`** owns the shell: renderer (ACES, PMREM RoomEnvironment, shadow frustum refit per scene size), camera fit/tween, showroom builder (flow layout, sections back→front: scenery → props/roads → vehicles), menu/pause/fullscreen/orientation, settings, boot, and the round lifecycle. **Cameras**: Orbit (default) and freecam — pointer-lock + WASD on desktop, fixed joystick + look-drag + ▲▼ on touch. Critical: **never run `controls.update()` while freecam or a scripted POV owns the camera** — OrbitControls re-aims at its target even when `enabled=false`. Crosshair targeting uses a per-round `Object3D → group id` map (rebuilt per round), raycast from screen centre in freecam or the pointer in orbit.
**Fog scale follows the LIVE camera during a round** (`syncFogScale`), not the distance `fitCamera` happened to pick: the metric is `fogBox.distanceToPoint(camera.position) / 8` clamped `fogFloor()`–20, against the round's bounding box cached at fit time. **The floor is not 1** — it is `groundRadius / 28` clamped 1.5–6, because the preset fog distances were authored when the world was a 90 m disc with nothing beyond it. At ×1 a dashcam saw 114 m and everything 1A and 1E build — hills, treeline, the whole landscape, which starts at 1.06 × playR — was a grey wall. The divisor is calibrated, not picked: preset far planes land near 75 m after a clear-weather boost and a scene of radius R wants ~2.5 R of visible world. On a 150 m arena that gives 400 m clear and 140 m in the `fog` kind. Distance to the box, **not to its centre** — that was the first version and it is wrong on exactly the scenes that matter, since a dashcam on a car at the far end of a 150 m highway is 120 m from the centroid while sitting 2 m from what it is filming, so it inherited the wide shot's ×20 and had its weather deleted. `distanceToPoint` is 0 anywhere inside the scene, so any camera among the cars gets weather at full strength. This is also what keeps weather **fair**: backing off far enough to read the whole scene is the same motion that pushes the fog away, so the wide read stays clear at every weather. `fitCamera` still seeds the scale from `dist` (where the camera is *going*) because mid-tween the camera is still back at the last shot. Rounds only — the showroom's free camera would otherwise thin its fog just by flying away from the display floor.

**Conventions everything relies on:**
- Vehicle space: forward = **+X**, ground = **y 0**, width along z. Builders return a `THREE.Group`; `buildVehicle()` recenters on x/z.
- Determinism: same seed+type+paint ⇒ identical vehicle. Type choice, build randomness and naming each use a *separately derived* rng (`makeRng('t:'+seed)`, `'b:'+seed+':'+typeId`, `'n:'+…`) so forcing a type doesn't shift the build stream. Never call `Math.random()` inside builders — always the passed `r`.
- **Merging (`lib.js` `bakeMerged` / `mergeByMaterial`)** flattens a STATIC subtree to one mesh per material. Sim-neutral by construction: colliders are explicit recipes never parsed from geometry, and fx reads `rec.spec`, not meshes. `byParams: true` keys on material *parameters* instead of identity — `matFactory` dedupes within one build, so 280 models ship 280 separate "black rubber" materials and identity-keyed merging can never join them. **Never merge a vehicle in the sim**: deform.js displaces per weld group with per-zone stiffness. The showroom is the one exception — nothing there is crashed. `userData.noMerge` opts a mesh out.
  - **Two different colour paths, and they are not the same operation.** `opt.color` BAKES each mesh's *material* colour into new vertex colours (1E's trick, collapsing a hundred one-off hexes onto one white material). Carrying an *existing* per-vertex colour attribute is the other case and is now automatic — it has to be, because `mergeByMaterial` never asked for colours, so geometry that already had them lost them silently: the material still says `vertexColors: true`, so the merged mesh renders against an undefined attribute and comes out black. Found in 1C when the merged riprap came back `hasColorAttr: false`. Same family as the indexed-geometry bug in 1H — plausible enough at a distance that nothing reports it. Precedence is mesh colours → material colour → white, so a mixed list can never go black.
- **The geometry kit (`lib.js`)** produces the look; prefer it over raw boxes. `slab(mat, {…})` is a chamfered-box frustum carrying `userData.pt`; `faceQuad(pt, side)` + `subQuad` + `quadPrism`/`panesOnQuad` place thin plates flush on any face including sloped ones (all windows, lights and grilles work this way). `faceQuad` assumes slab-shaped `pt` — on `wedge()` (plan-tapered) faces the panes skew, so use round `cyl` lamps there. `hexa(bottom4, top4)` is the underlying 8-corner solid; winding order is exact — copy an existing call rather than deriving anew. All geometry is non-indexed + `computeVertexNormals()`; materials come from the per-build `matFactory()` which dedupes by parameter key.
- **Family builders (`families.js`)** — `car()`, `truckFront()`+`chassis()`, `van()`, `bus()` — are config-driven; most of the 92 registry entries are parameter sets. Only genuinely novel shapes (F1, dragster, excavator, tram, bikes) get bespoke builders.
- **Registry (`vehicles.js` `REG`)**: `{ id, label, cat, build(r, M, ctx) }`. `ctx.paint` must win over any `paintHex` default. Adding a REG entry updates the UI, counts and smoke test automatically; optionally add a flavor name in `names.js`.

## Product rules (do not quietly change these)

- **Money**: integer dollars only; min bet $1; max = bankroll; payout = ⌊stake × odds⌋. Singles ×1.10–×50, parlay capped ×500. House margin ~6 % easing to ~2 % at d ≥ 8 — hard reads pay nearly fair. **Never move `marginFor` to make a gate pass**; that is changing the product to fit the test.
- **Anti-cheese**: campaign seeds hidden until settlement; each seed settles winnings **once per profile** (ledger); custom, shared and replayed seeds always run as **Exhibition** — playable, zero bankroll/ledger/stats/achievement mutation.
- **Difficulty** controls incident subtlety, cast size and similarity, camera coverage, near-miss (≥5) / decoy (≥6) / multi-incident (≥7) rates, and odds spread. **The freeze is offered at every level (P2/2H)** — it used to be withheld at d ≥ 8, which made the hardest rounds feel arbitrary rather than hard; the difficulty is the READ, not a denied study beat. Never unfair randomness: every outcome must be readable in the preview if you look at the right thing — d controls how hard "the right thing" is to spot. Campaign draws d on a geometric rarity curve (`P(d) ∝ 0.62^(d−1)`).
- **Persistence**: `localStorage["crashbet.profile.v1"]`, versioned JSON — bankroll, campaign cursor, current round + slip draft, settled-seed ledger (ring buffer, 500), stats, achievements, daily streak, settings. Boot always resumes the last unfinished round. Reset wipes after a confirm.
- Vehicles and property only. No people, no gore, ever.

## Gotchas learned the hard way

- **`box()` and `cyl()` return INDEXED geometry.** "All geometry is non-indexed" is true of the hexa/slab kit and false of three's primitives — `BoxGeometry` is 24 vertices + 36 indices, `CylinderGeometry` 52 + 96, `TorusGeometry` 45 + 192; only `IcosahedronGeometry` (`sphere()`) is non-indexed. Anything that reads a vertex buffer directly **must walk `geometry.index`**. Copying `position` in buffer order builds triangles from whatever vertices happen to be adjacent: it renders as long thin spikes reaching across the scene, and on small distant objects (vegetation) it is quietly wrong rather than obviously broken. `bakeMerged` is the one place this is handled — go through it.
- **A hidden subtree still costs a full matrix walk.** `visible = false` is a render-time cull; the renderer's own `scene.updateMatrixWorld()` recurses into hidden children regardless. The showroom's 7 321 nodes cost 2.3 ms on every frame of every round until it was *removed from the scene* rather than merely hidden (`showShowroom`). And `matrixWorldAutoUpdate = false` does **not** stop that recursion — in r169 it only guards whether that one node composes its own `matrixWorld` (`three.module.js:7778`). The per-node knob that actually saves work is `matrixAutoUpdate = false` (`freezeMatrices`).
- **Measure, then optimise, then measure again — this machine's frame timing has a ±1 ms noise floor.** A single A/B of the showroom detach read as "2.5 ms saved" from a change that was actually doing nothing; only an interleaved, warmed-up, median-of-N comparison separated the real 2.0 ms win from the noise. The plan's shadow-caster culling was measured this way and **rejected**: 499 casters, 60 % of them tiny, and disabling the shadow pass entirely changed nothing outside noise. Do not re-add it without a measurement that says otherwise.
- **Color math must stay in sRGB**: pass `THREE.SRGBColorSpace` to `getHSL`/`setHSL`. Linear-space HSL turns saturated reds salmon.
- Look-tuning baseline (don't casually change): exposure 1.18, key 1.7, hemi 0.55, fill 0.45; glass `#1b2836` rough 0.32 env 0.85. Brighter key lights wash colors through ACES. Weather multiplies these; it never replaces them, so a clear scene renders the baseline exactly.
- **You cannot darken a scene with the three lights alone.** `scene.environment` (PMREM RoomEnvironment) is a constant image-based flood that ignores every light intensity — the knob is `scene.environmentIntensity`. Symptom: killing the key to near zero yields a bright scene with black shadows instead of a dark one.
- `Mesh.position` is a read-only accessor — never `Object.assign(mesh, { position: … })`.
- Tall truck beds must pass `stacks: false` to `rigidTruck` or the exhaust stacks clip through the bed.
- Old vehicles must be disposed via `disposeGroup()` before replacement.
- `run.bat`'s port cleanup uses `findstr /c:` deliberately: a bare space in a `findstr` pattern means OR and would kill far too much.
- Physics wheel discovery keys off `userData.wheel` set in `parts.js` `wheel()` — a bespoke builder rolling its own wheel cylinders gets no suspension. Prefer `P.wheel`/`P.axle`.
- Prop bodies whose `node` IS the prop group must treat the node-local pose as identity in `_addPropRig` — reading `node.position` after the group was placed double-applies the offset (ramp colliders were 2× off).
- Rapier objects need explicit cleanup: `CrashSim.dispose()` frees world, event queue and controllers. Never drop a sim without it (wasm leaks silently).
- **A pinned scenario that bypasses the generator does not prove the generator works.** The causeway shipped a bridge over dry land for an entire phase because `generateScene` enumerated `world` keys and dropped `water`, while the `water` pin hand-wrote `world.water`. Likewise the `bridge` pin never caught the missing deck surface because its cars enter where deck y is exactly 0. `generateScene` must SPREAD `topo.world`, never re-list its keys — **and so must the `director` pin itself**, which re-listed `{world, roads, props, cars}` and would have silently dropped `junctions`, leaving the pin certifying a scene the game never builds. It is now `{ ...generateScene('pin-1', 4) }`.
- **Never compare crash-mode replays after N wall-clock frames — compare at the same TICK.** Auto slow-mo is on a 5-second real-time cooldown, so the same frame count executes different tick counts and hashes differ for reasons unrelated to determinism.
- **Never animate `opacity` from 0 with `animation-fill-mode: both` on anything load-bearing.** An animation that never starts holds its 0 % keyframe forever. Animate `transform` instead: a frozen scale degrades to "slightly small", a frozen opacity degrades to "the content does not exist".
- **`.mbtn` sets `width: 100%`, so it cannot go into a flex row unchanged** — `flex: 0 0 auto` resolves the basis to that 100 % and the first button eats the row. Add `width: auto` when flexing it.
- **A calibration correction must be measured as observed/expected against the model's own predictions**, never as a marginal rate — a marginal rate re-counts whatever the model's other keys already explain. And if the 95 % interval on your calibration sample is wider than the effect, the sample is too small to act on.
- The embedded Browser pane reports `document.hidden`, never fires `requestAnimationFrame`, and freezes CSS transitions and animations. Drive the game with `__app.pump(now)`; a "wrong" `getBoundingClientRect` may be a transition frozen mid-flight; and `computer{action:screenshot}` may time out (rasterize via `foreignObject` + a local sink instead). Do NOT conclude the sim/camera/HUD is broken because nothing moves on its own.
- **In that pane the `ResizeObserver` on `#stage` never fires either**, because observer callbacks are delivered as part of the rendering steps the pane does not run. So `resize()` only ever runs once, at boot, when `#stage` can still be 0×0 — and the canvas stays 0×0 while everything else works perfectly: the sim steps, `renderer.info` counts draw calls, `pump` returns true, and `toDataURL` hands back `"data:,"`. If a screenshot comes back blank, check `renderer.domElement.width` before suspecting the scene, then re-run what `resize()` does: `renderer.setSize(stage.clientWidth, stage.clientHeight)` + `camera.aspect` + `updateProjectionMatrix()`.
- **CSS edits need a hard reload in the preview pane** — the stylesheet is cached, so re-measuring right after an edit silently reports the OLD layout.
- **So do JS edits, and this one lies to you convincingly.** `await import('/js/roads.js')` from the console returns the instance already in the page's module registry, loaded when the page was — *not* what is on disk. `http-server -c-1` does not help; the registry is per-document. Injecting a probe built from a module you edited after the page loaded silently measures the OLD code: three roads with three different style bitfields came back with identical collider counts, which read as a broken opt-in and was a stale module. Hard-reload before any in-page probe of edited code.
- `?crash=N` boots with no user gesture, so WebAudio stays suspended until the first pointerdown — not a bug.
- `.claude/launch.json` uses `autoPort: true` (5183 is often held) — read the port from `preview_start`'s result rather than hardcoding it. `run.bat` still pins 5183 for manual use.
- **Never push unless explicitly told.**
