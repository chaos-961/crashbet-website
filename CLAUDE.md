# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"Crash Bet" — a deterministic crash-physics sandbox on top of a seeded procedural low-poly 3D vehicle generator (92 archetypes). Vanilla JS + Three.js r169 + Rapier 3D 0.19 (`libs/rapier3d-compat.module.js`, WASM inlined). No build step, no package.json, no framework. Three.js is vendored in `libs/` (import map in `index.html` maps `three` → `./libs/three.module.js`), so the app is fully offline.

**Current app shell (game era)**: boots to a main menu → "Start Game" goes fullscreen (landscape-locked on mobile, best effort) and opens the **showroom** — every model (92 vehicles · 51 scenery · 4 props · 2 road pieces) flow-laid on the ground disc, orbit/pan/zoom to browse. A top-left hamburger opens the pause menu (Resume / Settings: environment + quality / Fullscreen toggle / Leave). `?play=1` skips the menu without fullscreen (dev/headless). **The unified editor is mothballed, not deleted**: `js/editor.js` (spawn/gizmo/inspector/URL codec/worldgen UI) is intact but no longer imported by `main.js`, and its DOM was removed from `index.html` — re-wiring it needs its DOM back plus the `initEditor(ctx)` call. Editor-era URL params (`scene`/`car`/`prop`/`road`) are currently not parsed by the shell. The editor-era CSS is kept at the top of `css/style.css`; the game shell's styles are the marked section at the bottom.

## Running & testing

- Serve statically over HTTP (ES modules won't load from `file://`): `run.bat`, or `npx http-server -p 5183 -c-1 .` — then open http://localhost:5183. A `.claude/launch.json` config named `garage` exists for the browser preview.
- **Smoke test**: open `?smoke=1` — builds every registry type × 4 seeds at boot and logs `SMOKE DONE: N/N ok` (plus per-type stack traces) to the console. Run this after any change to `js/vehicles.js`, `js/families.js`, `js/parts.js`, or `js/lib.js`.
- **Determinism self-test**: open `?simtest=1` — runs five scenarios (`TEST_SCENARIOS` in physics.js: the legacy 3-car crash; an extended one with gravity 20, arena walls, ramp/boxes/pole props and per-car mass/grip/rest/crumple/delay/rolling/brake; a scenery one with seeded fixed+dynamic scenery props; a roads one with two spline roads whose curbs get clipped at speed; and a worldgen one that runs a full generated suburb — which also pins the generator, since layout drift changes the hash) twice each (300 fixed steps), FNV-hashes every body transform per step plus the final crumpled geometry, logs `SIM DETERMINISTIC: ok/FAIL`. Run after any change to `js/physics.js`, `js/deform.js`, `js/props.js`, `js/scenery.js`, `js/roads.js`, or `js/worldgen.js`. The hashes are stable across node and browser (same wasm f32 math); legacy reference `b09cc750`/geo `ff3b2b2e` must not drift unless physics intentionally changed.
- **Headless physics testing in node** (much faster than the browser loop): `node_modules/three/` is a gitignored shim so the whole vehicle+physics chain imports in node. Import modules via `pathToFileURL`, build a `CrashSim`, call `stepOnce()` in a loop, assert on `body.translation()` / `hashState()`. See the CrashSim API in `js/physics.js`.
- Syntax check without a browser: copy a module to `.mjs` and run `node --check`.
- **Headless visual verification**: `window.__app` exposes `{ renderer, scene, camera, controls, REG, env, fitCamera, invalidate, startGame, leaveGame, showroom }`. The pattern that works even when the preview tab reports `document.hidden` (rAF suspended, screenshots time out, canvas rect 0×0 — common in the embedded Browser pane on this machine): `await __app.startGame(false)`, then `renderer.setSize(w, h, false)`, position the camera, synchronously `renderer.render(...)` + `canvas.toDataURL()`, and POST the data URL to a local sink server that writes a PNG you can view. DOM overlays (menu/pause) can be rasterized headlessly via SVG `foreignObject` with the stylesheet inlined.

## Architecture

Module chain (each imports the previous): `js/lib.js` → `js/parts.js` → `js/families.js` → `js/vehicles.js` → `js/main.js` (+ `js/names.js`). Sim side: `js/deform.js` + `js/props.js` → `js/physics.js` → `js/editor.js` → `js/main.js` (+ `js/env.js`). Rapier is lazy-loaded (2.2 MB) — an empty boot never pays for it; it loads on first spawn or when a URL scene loads.

**Physics (`physics.js`) — determinism is the product.** Fixed 60 Hz timestep + accumulator; rendering interpolates between states and never feeds back. Zero `Math.random()` in sim code; world creation order is the backbone: ground → arena walls → props (array order) → cars (array order). Chassis colliders are convex hulls straight from each slab's `userData.pt` (+ big boxes/cylinders), rescaled to a per-category target mass (`CAT_PHYS`, scaled by footprint, × per-car `mass` multiplier) with a no-contact ballast collider that lowers the COM. Wheels come from `userData.wheel` tags (`parts.js` `wheel()`): duals dedupe to the outer wheel, inline bikes get invisible outriggers, tracked vehicles (dozer/excavator/roller/tram) get 4 virtual wheels at bbox corners. Raycast vehicle controller per car; front-cluster wheels steer, all wheels drive. Per-car tuning multipliers (`mass`/`grip`/`rest`/`crumple`) default to exactly 1.0 (or `rest: null` → 0.12) so legacy scenarios hash bit-identically. Launch phases in `stepOnce`: hold-with-brake until `delay`, then a `setLinvel` launch impulse (or none for `rolling` starts), full brake after `brake` seconds. Scenario shape: `{ world: {gravity, arena, walls}, props: [{kind,x,z,heading}], cars: [...] }`. "Perfect reset" = rebuild world + vehicles from the scenario (30 ms for 10 cars). **Incremental editing**: `replaceCar/appendCar/removeCarAt/setCarPose` (+ prop equivalents) touch only one object for paused-scene responsiveness — they do NOT promise a deterministic world, which is why the editor flags `dirty` and Play/Reset/Step always start runs from a full `build()`.

**Deformation (`deform.js`).** Geometry is non-indexed, so vertices weld by position (half-mm quantization) and displacement applies per weld group — faces never tear. Depth scales with contact Δv (impulse/mass, threshold 0.9 m/s), radial falloff + position-hash grain, accumulated crush clamped per group. Visual only — colliders stay rigid. Mesh↔chassis transforms are cached at build time (meshes never move relative to the wrap), so impacts convert world→mesh-local without touching render state.

**Editor (`editor.js`).** The whole authoring UX: spawn flow (pick → Spawn → tap ground), object chips, selection (hover Box3Helper, selected outline), a custom transform gizmo (object-local X/Z move arrows + rotation ring + free-drag on the body, mouse and touch), the inspector (Spawn / Selected / Scene `<details>` sections in `index.html`), keyboard shortcuts, and the URL codec. URL: `scene=g~slow~arena~walls~env`, repeated `car=seed~type~paint~x~z~deg~v~th~st~mass~grip~rest~crumple~delay~brake~rolling` (first 9 fields = legacy v1 format, still parsed; seeds get `~` force-escaped) and `prop=kind~x~z~deg`. Old garage `?seed=&type=&paint=` links spawn that single car. Editing seed/type/paint of a car calls `sim.replaceCar(i)` — only that car rebuilds. Every edit sets `dirty`; Play (when starting), Step and Reset flush it with a full deterministic rebuild. Limits: 10 cars desktop / 6 mobile. Inputs are recorded per-tick streams (`makeStream`) so constant launch params can later become real driving input.

**Roads (`roads.js`).** Catmull-Rom spline roads (centripetal, so tight zig-zags never cusp): control points are world coordinates, `buildRoad(spec)` sweeps an asphalt ribbon + geometry lane markings (white dashes or double yellow), raised curbs, optional sidewalks and continental crosswalks near open ends, and returns `{ group, shapes }` — shapes is a flat list of curb box-collider recipes that physics attaches to ONE fixed body at the origin (`_addRoadRig`; the group is never transformed). Zero randomness: segment counts derive from arc length, so the same spec is bit-identical. Strip winding rule: horizontals face up when `o0<o1`; the mirrored (s=−1) side negates offsets AND flips winding — pass offsets ascending for s=+1 or faces render invisible from above. Spec `{ pts, w, loop, style }`, style bitfield: 1 = double yellow, 2 = sidewalks, 4 = crosswalks. World build order is ground → walls → roads → props → cars (legacy scenarios have no roads, so their hashes never moved). Editor: `roadDraw` button lays points on ground taps (Enter/button finishes, Esc cancels), roads are selectable (orange point handles drag individual points, dragging the asphalt moves the whole road, width/style in the inspector rebuild via `sim.replaceRoad(i)`), URL `road=w~loop~style~x,z~x,z~…`.

**World generators (`worldgen.js`).** `generateWorld(preset, seed, {maxProps, maxRoads})` (presets: suburb / city / highway) emits plain scenario content — `{ world: {arena, env}, roads, props }` — from the existing road + scenery pipelines, so generated worlds are ordinary editable objects and the URL captures the whole result (loading a shared link never runs the generator). One rng stream per `'w:'+preset+':'+seed`; placement walks the road spline via `roadCurve`/`getPointAt` with lateral "zones" (sidewalk edge / front yard / lot) chosen so nothing overlaps by construction. Props are pushed identity-first (anchors → lots → filler) and trimmed to `maxProps`, so smaller (mobile) budgets drop filler, never the scene's identity — but note that means the same seed gives a *smaller subset* on mobile, and that trim order is part of each generator's design. Gotcha learned: the city loop's left normal points OUT of the block for its point winding — outside-ring props use positive lateral offsets, the plaza uses centroid math. Editor: "World generator" chips + seed in the Scene section replace roads+props (cars are kept; if there are none, one spawns at the road start), then do a full `rebuildSim`. Editor caps were raised for generated worlds: 48 props / 6 roads desktop, 20 / 3 on small screens (~390 draw calls at full desktop budget — a merge/instancing pass is the next lever if caps ever grow).

**Props (`props.js`) & environments (`env.js`).** Props return `{group, bodies}` with *explicit* collider recipes (never parsed from geometry): ramp + barrier are fixed single-body hulls (node === the group), box stack is 6 dynamic crate bodies, pole is one dynamic body with `y` rest height (origin at its center). Dynamic prop nodes get re-parented to `sim.root` and sync straight from their bodies. Environments are strictly visual (nothing the sim touches): preset palettes + deterministic decoration groups over the shared 90 m ground disc; `proving` reproduces the tuned lighting baseline exactly, others may scale light intensities. Fog near/far live in `env.state` and are scaled by `env.setFogScale(fk)` from `fitCamera` — don't set `scene.fog` directly.

**Conventions that everything relies on:**
- Vehicle space: forward = **+X**, ground = **y 0**, width along z. Builders return a `THREE.Group`; `buildVehicle()` recenters it on x/z, so builders never need to center themselves.
- Determinism: same seed+type+paint ⇒ identical vehicle. Type choice, build randomness, and naming each use a *separately derived* rng (`makeRng('t:'+seed)`, `'b:'+seed+':'+typeId`, `'n:'+...`) so forcing a type doesn't shift the build's random stream. Never call `Math.random()` inside builders — always the passed `r`.

**The geometry kit (`lib.js`)** is what produces the look; prefer it over raw boxes:
- `slab(mat, {x0,x1,y0,y1,w,wT,nose,tail,noseB,tailB,shiftT})` — a frustum defined by two rectangles (chamfered box). The returned mesh carries `userData.pt`, the parameter record.
- `faceQuad(pt, 'front'|'rear'|'left'|'right')` + `subQuad` + `quadPrism`/`panesOnQuad` — place thin plates *flush on any face of a slab, including sloped ones*. All windows, headlights, taillights, and grilles work this way (`P.headlightsOn(g, M, pt)` etc.). Caveat: `faceQuad` assumes slab-shaped `pt`; on `wedge()` (plan-tapered) faces the panes skew — use round `cyl` lamps there instead (see buggy/lemans).
- `hexa(bottom4, top4)` is the underlying 8-corner solid; winding/order is exact — copy an existing call rather than deriving anew.
- All geometry is non-indexed + `computeVertexNormals()` for flat shading; materials come from the per-build `matFactory()` (`M(hex, opts)`) which dedupes by parameter key.

**Family builders (`families.js`)** — `car()`, `truckFront()`+`chassis()`, `van()`, `bus()` — are config-driven; most of the 92 registry entries are just parameter sets. Only genuinely novel shapes (F1, dragster, excavator, tram, bikes…) get bespoke builders in `vehicles.js`.

**Registry (`vehicles.js` `REG`)**: `{ id, label, cat, build(r, M, ctx) }`. `ctx.paint` is the user's paint override and must win over any `paintHex` default (`bodyHex: ctx.paint || ...`). To add a vehicle: add a REG entry (UI dropdown, counts, and smoke test pick it up automatically), optionally add a flavor-name entry in `names.js` keyed by id.

**`main.js`** owns the game shell: renderer (ACES tone mapping, PMREM RoomEnvironment, shadow frustum refit per scene size), camera fit/tween, the showroom builder (flow layout: build → rotate to face +Z → measure bbox → wrap rows of ≤100 m; sections back→front are scenery → props/roads → vehicles), menu/pause/fullscreen/orientation flow, settings (env preset + Low/High quality: DPR 1 + shadows off vs full), Esc = pause, and boot (`?smoke`, `?simtest`, `?sheet`, `?play=1` = straight into the showroom without fullscreen). **Cameras**: Orbit (default) and **freecam** — pause-menu Camera chips, HUD 🎥 button, or `C`. Freecam is a true fly cam: pointer-lock mouse look + WASD (Space/Ctrl or E/Q vertical, Shift ×3, wheel = speed 5–140) on desktop; fixed left joystick + right-half look-drag + ▲▼ altitude buttons on touch (`#flyui`, shown via `body.ingame.flycam` + coarse pointer). Velocity is lerp-smoothed; `flyUpdate` returns false at rest so render-on-demand still sleeps. Critical: never run `controls.update()` while freecam owns the camera — OrbitControls re-aims at its target even when `enabled=false`. **Skybox** (`env.js` `SKIES`/`makeSky`): per-preset camera-following gradient dome + sun/moon disc + deterministic stars/cloud blobs, all `fog:false, toneMapped:false`; `scene.fog` is tinted to the sky horizon (the flat `bg` era is over; the tuned *lighting* values are unchanged) and `env.syncSky(camera.position)` runs every frame in the loop. **`editor.js`** (dormant) still holds the whole editor era: spawn flow, gizmo, inspector, URL codec, worldgen UI — see "Current app shell" above for what re-wiring needs.

## Gotchas learned the hard way

- **Color math must stay in sRGB**: pass `THREE.SRGBColorSpace` to `getHSL`/`setHSL` (see `shade`/`jitterColor` in `lib.js`). Linear-space HSL turns saturated reds salmon.
- Look-tuning baseline (don't casually change): exposure 1.18, key light 1.7, hemi 0.55, fill 0.45; glass `#1b2836` rough 0.32 env 0.85. Brighter key lights wash colors through ACES.
- `Mesh.position` is a read-only accessor — never `Object.assign(mesh, { position: ... })`; use `mesh.position.set(...)`.
- Tall truck beds (box/garbage/army/mixer) must pass `stacks: false` to `rigidTruck` or the exhaust stacks clip through the bed.
- Old vehicles must be disposed via `disposeGroup()` (in `lib.js`) before being replaced — `generate()` already does this.
- `run.bat`'s port cleanup uses `findstr /c:` deliberately: a bare space in a `findstr` pattern means OR and will match (and kill) far too much.
- Physics wheel discovery keys off `userData.wheel` set inside `parts.js` `wheel()` — a bespoke builder that rolls its own wheel cylinders won't get suspension (it'll fall back to virtual wheels). Prefer `P.wheel`/`P.axle`.
- The crash sim's visual wheel groups get re-parented (`wrap.attach`) at rig build so the controller can drive them; plain `buildVehicle()` builds (smoke test, contact sheet, .glb export of an unspawned car) are untouched.
- Prop bodies whose `node` IS the prop group must treat the node-local pose as identity in `_addPropRig` — reading `node.position` after the group was placed double-applies the prop offset (learned the hard way: ramp colliders 2× off).
- The embedded Browser pane freezes CSS transitions when `document.hidden` — a "wrong" `getBoundingClientRect` may just be a transition frozen mid-flight; disable the transition before measuring.
- Rapier objects need explicit cleanup: `CrashSim.dispose()` frees the world, event queue and controllers — never drop a sim without calling it (wasm memory leaks silently).
