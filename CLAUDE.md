# CLAUDE.md

Guidance for Claude Code (claude.ai/code) working in this repository.

## What this is

**Crash Bet** — a betting game about reading physics, built on a deterministic crash sim and a seeded procedural low-poly world. Every round deals a generated traffic scene ten seconds from disaster, the player bets on what the chaos will do, and the sim settles every bet with zero ambiguity. Vehicles and property only — no people, ever.

Vanilla JS + Three.js r169 + Rapier 3D 0.19, both vendored (`libs/`, import map in `index.html`). No build step, no package.json, no framework, fully offline, installs as a PWA. **The game is feature-complete and not deployed.**

**App shell**: boots to a main menu gated behind a boot preloader (physics + Rapier wasm + fx pools + showroom warm up behind a progress bar; `preloaded` flips at 100 %). Menu: **Continue/Resume** (campaign) · **Daily** (one date-derived scene a day) · **Stats** (record + 16 achievements) · **Custom seed** (Exhibition) · **Garage** (showroom of all 280 models) · **How to play** · **Crash test** (8 seeded set-piece wrecks) · **New run**. In game: hamburger → pause menu (Resume / Settings: environment · quality · camera · volume · motion / Fullscreen / Leave). Dev URL params: `?scene=<seed>~<d>`, `?crash=N`, `?play=1`, `?smoke=1`, `?simtest=1`, `?sheet=1` (vehicles) / `?sheet=scenery` (all 184 scenery models).

**The editor is mothballed, not deleted**: `js/editor.js` (spawn/gizmo/inspector/URL codec/worldgen UI) is intact but not imported, and its DOM was removed from `index.html` — re-wiring needs that DOM back plus `initEditor(ctx)`. Editor-era URL params (`car`/`prop`/`road`) are no longer parsed. Editor-era CSS is kept at the top of `css/style.css`; the game's styles are the marked section below it.

## Running & testing

- Serve over HTTP (ES modules will not load from `file://`): `run.bat`, or `npx http-server -p 5183 -c-1 .`. `.claude/launch.json` has a `garage` config for the browser preview.
- **Smoke test**: `?smoke=1` — builds every registry type × 4 seeds, logs `SMOKE DONE: N/N ok`. Run after changing `vehicles.js`, `families.js`, `parts.js`, `scenery.js` or `lib.js`. Currently **1104/1104** (92 vehicles + 184 scenery, × 4). It is a coverage count, not a pin — it moves whenever the registries grow.
- **Contact sheets**: `?sheet=1` tiles every vehicle, `?sheet=scenery` every scenery model, into one canvas at `window.__sheet`. Both run synchronously at boot so they work with rAF suspended. With 184 scenery models this is the only practical way to review a batch, and reviewing a batch is how backwards winding, floating parts and wrong-axis rotations get caught — every defect in the §1D batch was found this way and none of them failed the smoke test.
- **Determinism self-test**: `?simtest=1` — runs every `TEST_SCENARIOS` entry in physics.js twice (300 fixed steps), FNV-hashes every body transform per step (debris included) plus the final crumpled geometry. Run after changing `physics.js`, `deform.js`, `props.js`, `scenery.js`, `roads.js`, `worldgen.js` or `director.js`. Hashes are stable across node and browser (same wasm f32 math).
- **Pin gate**: `node tests/simpins.mjs` — `?simtest=1` only proves run-to-run stability; this proves the values never MOVED, which is the invariant that actually protects the sim. **Current pins (10)**: legacy `769824bd`/geo `f11ed37a` · extended `54489fae`/`20413f1b` · scenery `50fa7593`/`ec076bdc` · roads `4ac04ea9`/`ab859269` · deck `354b55b0`/`f0c5be44` · bridge `41c337c8`/`af80eb9` · water `ac116fb4`/`9d559754` · carnage `98c34be0`/`584ce83a` · director `d4276a3c`/`44620267` · worldgen `20bf2a4d`/`418447b5`. None may drift unless physics/deform intentionally changed — **update this line when they do**, and record why in the file next to the pin.
- **Scrub gate**: `node tests/scrub.mjs` — the freeze scrub rewinds by re-simming from t0, so this asserts a scrub round-trip lands on the EXACT state a clean run reaches (per seed × difficulty, rewinding to arbitrary ticks including 0). A money-correctness gate, not a UI one: the slip settles against a recording whose incident starts at `INCIDENT_TICK`, so a sim resuming from a different state plays a scene the odds were never priced on.
- **Economy suite**: `node tests/economy.mjs` — payout flooring, bankroll floors, once-per-seed ledger, parlay caps, mid-round resume, the Exhibition rule, the daily seed (shared across profiles, one paying attempt, streak arithmetic across month/year boundaries), achievements (idempotent, Exhibition unlocks nothing, a throwing test cannot break settlement), plus a real scene → markets → settlement integration pass. Currently **56/56**.
- **Weather gate**: `node tests/weather.mjs` — weather never touches the sim, so no pin can catch a mistake in it. What this actually guards is the silent-fallback class: `WEIGHTS` falls back to `proving` on an unknown env id, so it asserts `ENVS` and `WEATHER_ENVS` agree **in both directions** (a preset added without a table would inherit a desert's weather, and nothing would say so — precisely how `city` went unnoticed). Plus descriptor invariants, roll determinism, grid staying clear, and ≥4 kinds per env because "every scene on a preset looks identical" is the problem weather exists to solve. Currently **21/21**. Run after touching `weather.js` or `ENVS`.
- **Scene sweep**: `node tests/sweep.mjs [seedsPerD]` (default 60 → 180 scenes) — deals across d 1/5/10, records to rest, reports coverage per topology × template. Hard defects: a car↔car hit before the incident tick, no markets, a missing headline pair. Quality *budgets* (rates, not per-scene defects): ran-to-cap ≤ 10 %, eventless ≤ 12 % — both are legitimate outcomes, so they are tracked with headroom rather than pretended away. The pre-incident check counts **car↔car only**: the recorder logs kerb and ground contacts as `hit` too, and clipping a kerb on the approach is not a choreography failure.
- **Monte Carlo gate**: `node tests/montecarlo.mjs [roundsPerD]` (default 100 × 10 ≈ 9 min) — bets every market to-win-$100 and asserts the realized house edge sits inside the per-difficulty band around `marginFor(d)`. Seeds are fixed (`mc…`), so **re-running the same size is deterministic and cannot tell you whether a band miss is noise** — vary `roundsPerD` to resample. Currently **10/10 bands, overall edge 2.45 %** over $913 k staked. Calibration seeds (`cal…`) and gate seeds (`mc…`) are disjoint, so the measurement is genuinely out-of-sample. Beware short runs: at 15 rounds/d the estimator's σ is ≈ 7 %, so a small sample "fails" on noise alone.
- **Odds calibration**: `node tools/calibrate.mjs [n]` prints the empirical CALIB table to paste into `js/markets.js`. Regenerate whenever director/physics changes move outcome statistics. It also prints, to stderr, observed/expected per difficulty band with a 95 % interval — that is the number the Monte Carlo actually tests (edge ≈ `1 − (1 − margin)·O/E`), and it is far more diagnostic than reading edge percentages afterwards.
- **Headless physics in node** (much faster than the browser loop): `node_modules/three/` is a gitignored shim so the whole chain imports in node. Import via `pathToFileURL`, build a `CrashSim`, call `stepOnce()`, assert on `body.translation()` / `hashState()`.
- Syntax check without a browser: copy a module to `.mjs` and run `node --check`.
- **Headless visual verification**: `window.__app` exposes `{ renderer, scene, camera, controls, REG, env, fitCamera, invalidate, startGame, leaveGame, setCamMode, fly, flyUpdate, showroom, startCrash, nextCrash, replayCrash, crash, crashFx, startScene, resumeRound, seekPreview, round, preloaded, pump }`. **`__app.pump(now)` runs one real frame** (camera tweens, sim, fx, HUD, render) with a synthetic clock — loop it with `now += 1000/60` to drive the game headlessly instead of re-implementing the loop. DOM overlays can be rasterized via SVG `foreignObject` with the stylesheet inlined (serialize with `XMLSerializer` — raw `innerHTML` is not valid XML).

## Architecture

Module chain (each imports the previous): `lib.js` → `parts.js` → `families.js` → `vehicles.js` → `main.js` (+ `names.js`). Sim: `deform.js` + `props.js` → `physics.js` → `main.js` (+ `env.js` + `fx.js`); `terrain.js` is pure and imported only by `env.js`; `weather.js` imports only `lib.js` and is consumed by `main.js` (env.js takes a weather *descriptor*, never the module). Game: `director.js` (imports roads + worldgen, NEVER physics) → `recorder.js` (imports physics + director + roads) → `main.js`; physics.js imports director.js only for its pinned `director` scenario. `markets.js` → `economy.js` → `betui.js`. `achievements.js` is pure and imports nothing. `editor.js` is dormant. Rapier is lazy-loaded (2.2 MB) during the boot preloader.

**Physics (`physics.js`) — determinism is the product.** Fixed 60 Hz timestep + accumulator; rendering interpolates and never feeds back. Zero `Math.random()` in sim code; world creation order is the backbone: ground → arena walls → roads → props → cars (array order). Chassis colliders are convex hulls from each slab's `userData.pt`, rescaled to a per-category target mass (`CAT_PHYS`) with a no-contact ballast collider lowering the COM. Wheels come from `userData.wheel` tags; duals dedupe, inline bikes get invisible outriggers, tracked vehicles get 4 virtual corner wheels. Raycast vehicle controller per car. Per-car tuning multipliers default to exactly 1.0 (`rest: null` → 0.12) so legacy scenarios hash bit-identically. Scenario shape `{ world: {gravity, arena, walls, water}, roads, props, cars }`; "perfect reset" = rebuild from the scenario (~30 ms for 10 cars). **Incremental editing** (`replaceCar`/`appendCar`/…) touches one object for paused responsiveness and does NOT promise a deterministic world — that is why the editor flags `dirty` and Play/Reset/Step always start from a full `build()`. **Damage model**: `CAT_PHYS` carries per-category `sus`, `aDamp`, `crumpleK`, `wheelTough`; impacts charge nearby wheels, past `WHEEL_BENT_AT` a wheel steers crooked, past `WHEEL_DETACH_AT` it tears off into a real dynamic debris body inheriting `v + ω×r` (`hashState` folds debris in; any new debris path MUST register with `this.debris` or cleanup and hashing miss it). Six render-side hooks — `onImpact/onScrape/onGlass/onDetach/onSplash/onSunk` — fire inside `stepOnce`; **the only sanctioned write-back is `sim.speed`** (wall-clock pacing).

**Deformation (`deform.js`).** Non-indexed geometry, vertices weld by position (half-mm quantization), displacement applies per weld group so faces never tear. Depth scales with contact Δv, radial falloff + position-hash grain, accumulated crush clamped per group. Visual only — colliders stay rigid. Per-group zone stiffness (nose/tail crumple deep, cabin resists, roof gives in rollovers), work hardening, ridged crease noise + tangential shear, and two-stage glass (crack → shatter) driven by direct impact overlap plus inverse-square "frame shock" that pops windshields from bumper hits.

**Roads (`roads.js`).** Catmull-Rom spline roads (centripetal, so tight zig-zags never cusp). `buildRoad(spec)` sweeps asphalt + geometry lane markings, curbs, optional sidewalks and crosswalks, returning `{ group, shapes }` — shapes are curb box-collider recipes physics attaches to ONE fixed body at the origin (the group is never transformed). Zero randomness: segment counts derive from arc length. Strip winding rule: horizontals face up when `o0<o1`; the mirrored side negates offsets AND flips winding — pass offsets ascending for s=+1 or faces render invisible from above. Style bitfield: 1 = double yellow, 2 = sidewalks, 4 = crosswalks, 8 = bridge deck.
**Elevation** is opt-in and defaults to exactly 0, which is why every pre-elevation hash survived it: Catmull-Rom interpolates components independently so all-zero y gives exactly 0 out, and every height is an offset ABOVE the frame's surface (`f.y + y0`) so `+ 0` is a bit-level no-op — do not "simplify" that to absolute heights. Tangents normalise by HORIZONTAL length, so the ribbon never narrows as it climbs. Curb colliders keep the yaw-only quaternion on flat roads and only pay for the pitched basis when `isElevated(spec)` — routing flat roads through a matrix→quaternion conversion would perturb the last bits. **Elevated runs also emit a driving-surface slab** (opt-in on `elev`): without it the deck is decoration and a car mid-span falls straight through. Its longitudinal overlap is deliberately much smaller than the curbs' — consecutive slabs pitch to their own tangent, so a generous overlap leaves the next slab's leading edge proud, a lip that trips a wheel at speed. Nothing drives on a curb; something drives on this.

**Water (`physics.js` `_stepWater` + `env.setWater`).** Opt-in via `world.water`; null for pre-water scenarios so the path is skipped entirely. The ground stays a single 220 m slab UNLESS a basin exists, in which case it becomes four land slabs around the hole plus a bed collider — and **those land slabs are 20 m thick on purpose**: at 2 m a submerged car slid out from under them and fell forever. Submerged bodies get saturating buoyancy (0.62 g, deliberately under neutral so wrecks settle instead of bobbing) plus linear and angular drag. Determinism rules match the driver controller: arithmetic only, and thresholds compare **squared** magnitudes — never `Math.hypot`, whose last bits are not guaranteed across engines. `env.setWater` is purely visual and is cleared in `destroyRound`.

**Scenery (`scenery.js`) — 184 kinds in 8 categories.** `{ id, label, icon, cat, build(r, M) → {g, bodies} }`, same contract as `vehicles.js`: same kind+seed ⇒ identical model AND identical collider recipe, zero `Math.random()`. Categories: Nature · Suburbia · Street & City · Signs & Traffic · Water & Coast · Mountain & Alpine · Rural & Farm · Industrial. **Adding entries is safe by construction** — the generators pick kinds from hardcoded dressing lists, so new models are inert until something names them, while the registry auto-propagates to showroom, counts, crosshair labels, the smoke test and the contact sheet. Heavy things use `fixedBody`, knockable things `dynGround` (which re-roots visuals so the body origin sits at H/2). Traps worth knowing:
- **`cyl(mat, {axis: 'x'|'z'})` IS a rotation** — it sets `rotation.z = -π/2` (or `rotation.x = π/2`). Writing `.rotation.set(…)` or assigning that same component afterwards silently *undoes* it: a felled tree stood back up and beached driftwood turned into fenceposts, and neither failed the smoke test. Only `.rotation.y` is safe to assign on an axis-built cylinder; for anything more, wrap it in a group. The roof idiom (`rotation.x = π/2; rotation.z = π/4` after `axis:'z'`) overrides deliberately and is fine.
- **Nothing batches, so every mesh is a draw call.** A three-high container stack came out at 177 meshes — 3.5× the heaviest building — purely from corrugation ribs. Distance-read objects get an explicit LOD flag (`containerBody(…, near)`); the library's ceiling is ~68 meshes (`transmission_tower`, an irreducible lattice).
- **You cannot subtract one solid from another here.** An open hull, a bay, a trough has to be built as a *shell* — floor plus canted sides plus ends. Nesting a dark slab inside a light one to fake an interior just hides the dark one.
- **Silhouette beats surface.** Crags that only dent a face leave a box reading as a box; they have to break the outline, which means pushing past the top edge and off both ends.

**Props (`props.js`) & environments (`env.js`).** Props return `{group, bodies}` with *explicit* collider recipes, never parsed from geometry. Dynamic prop nodes re-parent to `sim.root` and sync from their bodies. Environments are strictly visual: preset palettes + deterministic decoration over the shared ground. **`ENVS` is 5** — `proving · salt · night · grid · city`; `city` exists because `topoIntersection` and `topoRoundabout` have always asked for it (`director.js:267, 478`) and always silently got `proving`, since `apply()` falls back on an unknown id. The fix was to add the missing preset, never to edit the director — that keeps every generated scenario byte-identical. **An env id a topology names but `ENVS` lacks fails silently**, so adding topologies means checking both ends. Fog near/far live in `env.state` and are scaled by `env.setFogScale(fk)` — don't set `scene.fog` directly. **Skybox**: per-preset camera-following gradient dome + sun/moon + deterministic stars/clouds, all `fog:false, toneMapped:false`; `env.syncSky(camera.position)` runs every frame.
**The preset/weather split** is what makes per-scene variation expressible at all: `apply(id)` rebuilds STRUCTURE (sky mesh, decoration, terrain, ground texture, light *colours*) and early-returns on an unchanged id; `applyWeather(w)` moves intensities, fog distances, haze and cloud cover, and **never** early-returns. The old single `apply` did both and bailed when the id matched, so "same preset, different weather every scene" could not be said. `apply` ends by calling `applyWeather` so a preset change re-lands the current weather rather than dropping it.

**Terrain (`terrain.js`) — and it IS the ground.** The landscape around *and under* the play area, owned by env (`env.setTerrain(spec)` / `env.terrainField`), driven from `world.terrain` or a bare seed. It reaches r=0, so while it is up the old flat `CircleGeometry` disc **hides rather than being deleted** — the showroom and crash mode still want a plain apron, and a hidden mesh costs nothing. Inside `playR` the height mask is exactly 0, so the play area is the same flat y=0 plane the disc was, receiving shadows the same way; what changed is that the surface colour is a radial ramp **in world units** instead of a stretched CanvasTexture (which also retires the bug where `setGroundRadius` rebuilt the geometry but never rescaled the texture, so one gradient stretched identically over 90 m or 167 m), and the world no longer ends at a visible rim. **Ring 0 collapses to the origin**, so its second triangle per quad is zero-area — emit it and `computeVertexNormals` hands back a zero-length normal and a black speck at the scene centre; ring 0 is a triangle fan and the vertex buffer is trimmed to what was actually written. **Opt-in with null exactly like `world.water`** — no spec, no mesh, no cost, and that is why every sim pin survived it. One polar mesh, never a near disc plus a far ring: `rings × spokes` with radii packed as `u^1.7` so detail lands just outside `playR` where the eye is. Displacement is masked by `smoothstep(playR, playR*1.35, r)`, so it is *exactly* 0 on the drivable area — terrain can never intrude on a lane and there is no seam to hide. Height is integer-hash value-noise fbm with an optional ridge fold (`1-|2n-1|`, squared) and a domain warp, because unwarped ridgelines run along the noise grid and read as corduroy. Vertex colours (rock/grass/scree/snow by height and slope), non-indexed + `computeVertexNormals()` for the faceted read, one `MeshStandardMaterial`, one draw call. `makeHeightField(spec)` is exported separately so vegetation scatter and anything else that must sit ON the landscape queries the same surface the mesh was built from. Three things that cost real time:
- **Winding is the same trap as roads.js.** Spoke index runs with +angle and ring index with +radius, so the natural `(00,01,11)` order yields a −y normal and the entire landscape is backface-culled from above — present, lit, invisible.
- **Amplitudes are set against this world's scale, not the real one.** Play area ~90–170 m against a 430 m dome means hills only read between roughly 30 m and 120 m; at 20 m a ridge 400 m out subtends nothing and the ring renders as a flat wash. The far band gets a ×3.6 boost specifically to break the skyline.
- **A polar grid stripes.** Every vertex on a ring sharing one radius aligns height contours with the rings, and flat shading turns that into concentric bands down every hillside. Each vertex is jittered within its own cell (ring 0 pinned, since it meets the ground disc) — topology unchanged, lattice broken.

**Weather (`weather.js`).** Per-scene weather, strictly render-side like fx.js: it reads nothing from the sim and writes nothing back, and Phase 1 keeps it out of physics entirely (grip is a P2 opt-in on `world.weather.grip`), which is why no pin moved. `rollWeather(seed, envId)` returns a plain descriptor — no THREE objects, loggable and testable — from its OWN rng stream (`'wx:'+seed`), so the same seed always shows the same sky and no existing stream shifts by a draw. 11 kinds, weighted **per environment** because weather has to fit the place (a desert blizzard reads as a bug, and `grid` stays clear so the diagnostic preset stays legible). main.js rolls it in `startScene`, hands the descriptor to `env.applyWeather(w)` and the particles to `weather.set(w)`; `destroyRound` clears both so the rain does not follow you into the menu.
- **The lighting is the feature; the particles are garnish.** An overcast scene reads as overcast with no rain drawn at all. `key/hemi/fill/exposure/envI` are MULTIPLIERS over whatever the preset authored, so weather composes with a place instead of replacing it.
- **`envI` (`scene.environmentIntensity`) is the biggest lever and the easiest to miss.** The PMREM RoomEnvironment is a constant IBL flood that no light intensity touches; leave it at 1 and a storm on a pale preset is a bright afternoon that merely lost its shadows. Wired through `ctx.setEnvIntensity`, next to `ctx.setExposure` — and `BASE_EXPOSURE` (1.18) stays the look-tuning baseline that weather multiplies, so a clear scene is bit-for-bit what the game was tuned against.
- **`hemi` goes DOWN in bad weather, not up.** The instinct is to raise it — overcast really is a sky-wide source, the shadows really do go away — and the first table went to 1.38. On a white salt flat that produced a storm you could read a newspaper by: albedo wins, and killing the key alone cannot darken a surface the ambient is still flooding. It rises only where the ground genuinely bounces (snow, mist). The key:hemi *ratio* still climbs, which is what actually sells flat weather light.
- **Haze is a colour, not a grey.** Each kind carries `haze {hex, amt}` — dust is tan, mist is *brighter* than what it hides, a storm is near-charcoal — and the sky gradient, `scene.background`, the fog tint and the terrain's baked horizon must all derive from the same `hazeTarget(preset, haze)` expression or the landscape meets a sky it does not match. Amount is `max(cloud, fog×0.95)`: cloud alone under-reads dust, which hazes hard on a half-clear sky. Cloud and haze are separate knobs on purpose — cloud hides things *in* the sky (sun, stars, blob count), haze recolours the air, and driving blob count off haze grows clouds out of a dust storm.
- **Wet roads (`applyWetness`) do more work than the rain.** A downpour falling on bright dry asphalt reads as a bug and no particle density fixes it. Roads only — materials come from a per-build `matFactory` cache so nothing leaks into the next round, and terrain is left alone (grass and rock do not gloss). Scalar-multiplying a linear-space `m.color` is an exposure change, not an HSL round-trip, so it is exempt from the sRGB rule.
- **Precipitation is two pooled systems with zero per-frame CPU** — fall, wind drift and wrap all happen in the vertex shader against a 44×30×44 box that follows the camera, so changing the amount is a draw-range write. The box is deliberately small: density is count/volume, and a box big enough to "cover the scene" spread an affordable drop count into six visible streaks. Rain is `LineSegments` because `gl_PointSize` is square and cannot express a velocity-aligned streak. **Flakes need ~2.5× the drop count for the same read** (a drop crosses many pixels, a flake is a dot; at rain counts snow was ~20 specks and read as a dirty lens). Whole system costs **one draw call**.
- **Flakes size in metres through the projection** (`aSize × uPix / -mv.z`, `uPix = h / 2tan(fov/2)`, refreshed in `resize`) — sizing in raw pixels makes near flakes 200 px ping-pong balls. Correct sizing alone still leaves the one flake that drifts within a metre of the lens rendering as a fat soft ball, so there is also a `smoothstep(0.7, 3.5, z)` near-fade; no real lens resolves what is inside its near focus.
- Weather is **frozen while the round is** (`update(dt, camera, paused)`): the sim's clock is stopped so hanging rain is the correct fiction, and it lets render-on-demand sleep through the freeze — the longest UI phase, and exactly when the player wants a steady frame to read.

**World generators (`worldgen.js`).** `generateWorld(preset, seed, {maxProps, maxRoads})` (suburb / city / highway) emits plain scenario content, so generated worlds are ordinary editable objects. One rng stream per `'w:'+preset+':'+seed`; placement walks the spline with lateral zones chosen so nothing overlaps by construction. Props push identity-first (anchors → lots → filler) and trim to `maxProps`, so smaller budgets drop filler, never identity — but note the same seed gives a *smaller subset* on mobile, and that trim order is part of each generator's design. Gotcha: the city loop's left normal points OUT of the block, so outside-ring props use positive lateral offsets and the plaza uses centroid math.

**Effects (`fx.js`).** Strictly render-side; reads sim state and hooks, never writes back (fx has its own `makeRng('fx')`). Pooled zero-alloc systems: four shader point clouds (sparks / normal-blend puffs / fire / water spray), instanced glass shards and paint chips, a ground-decal ring buffer (skids, scrape grinds, fluid spots — deterministic y micro-offsets kill z-fighting), an impact flash PointLight whose colour is set per trigger, and camera shake applied around `renderer.render` only then undone. Struck props start *standing* emitters keyed on `props[i].spec.kind` — geysers (hydrant), arcing sparks (poles/lamps/signals) — anchored to the prop's SPEC position, not its group, because these props are dynamic and the body tumbles away while the sheared main stays planted. Fire spread is render-only ignition in an fx-local map (`frontDmg` is sim state and fx must never write it) and only a shell already past `damage > 9` catches. `initFX(scene, {small})` halves pools on small screens; `attach(sim)` wires the hooks and re-keys `perCar`; `update(dt, camera)` returns busy so render-on-demand still sleeps when idle.
**Audio (`Sfx`)** is procedural and zero-asset. The engine bed is a second sustained voice alongside the scrape loop, which works only because it occupies a different band: two detuned saws at 55–190 Hz under a 340 Hz lowpass vs the scrape's 1050 Hz bandpass. UI sounds go through `_ping`, which deliberately does NOT consume a crash one-shot slot — UI feedback must not be dropped because six cars collide in one frame. Volume lives on the master gain; `mute()` restores to the saved volume. The context needs a user gesture, so `?crash=N` boots register a one-time pointerdown unlock.

**Incident engine (`director.js`).** `generateScene(seed, d)` deals a complete deterministic round from separately derived rng streams per facet (`'scn:'+seed+':topo'/'incident'/'cast'/'dress'`). 10 topologies (bespoke signalized intersection + suburb/city/highway from `generateWorld` + causeway · switchback · schoolzone · tramcrossing · parkinglot · roundabout); 20 incident templates, each returning cars with per-tick `drive` specs (polyline + cruise speed + sparse `cmds`) and optional props. **Everything goes wrong at exactly tick 600.** Hard-won invariants:
- **Nothing may collide before tick 600.** A conflict scrub staggers crossing arrivals, then a spawn-overlap net runs BOTH before and after the scrub, since the scrub moves cars.
- Head-on templates (`needsOpp`) only deal where opposite-lane geometry exists, and the victim's arrival window accounts for COMBINED closing speed (windows sit at ~T+70–120).
- Swerve cmds that must leave the lane need `off: true` or pure pursuit cancels the bias. Heavies are barred from fast bendy plans (`v²·κ > 2.4`).
- Dressing is scrubbed off lane corridors, out of big-prop footprints (a tree inside a house collider gets ejected and falls across the road), and away from car spawns.
- **`sortLanes` quantises its length key to cm and tie-breaks on build order.** The two directions of a symmetric road are equal to within float noise, and a raw `b.len - a.len` let `sqrt`'s last bits decide — node and the browser sorted a causeway's lanes oppositely and **generated mirrored scenes from one seed**, which breaks the core promise that the pre-sim the bets settle against is the scene the player watched.
- **`place()` may drop below `MIN_V` to hit its arrival tick exactly.** Lanes are pre-filtered for a 600-tick run, but templates aiming past T=0 need more road; the old floor clamped `s0` to the lane start so the car arrived EARLY and rear-ended a still-parked victim at tick 599.
- **The near-miss shift is capped by the room the victim can give up** and still reach its anchor after T=0. Shifting it up-lane also makes it arrive sooner; unbounded, that silently undid the arrival budget in both directions. The spawn-overlap net cannot catch this class — the cars never overlap at spawn, they converge early.

**Driver controller (in `physics.js`).** Cars with `spec.drive` get closed-loop pure pursuit: arithmetic-only (`+,-,*,/,sqrt,min,max,abs` — no transcendentals, so bit-deterministic across engines), anticipatory `v² = 2as` braking, `end: 'stop'|'coast'`, sparse cmd timeline, post-crash shock (damage > 6 → brake and hold), and **ACC car-following for ambient cars**: path-probe detection (goal + far + midpoint — a straight-ahead cone misses leaders around bends), a same-direction gate (dot > 0.2, since reacting to crossing traffic shifts junction timing and breaks the conflict scrub), capping to the leader's TRUE velocity projected on own heading, full attention only before the incident tick. Driven cars get real brake authority (`brakeK` 0.055 vs the legacy 0.02, which keeps old hashes frozen). `sim.stopAt` freezes on an exact tick; `colToObj` maps collider handles → `{kind}` for impact attribution.

**Pre-sim recorder (`recorder.js`).** `recordScene(...)` runs the scenario headlessly to rest and returns `{events, summary, restTick, ticks, hash, tracks}` — the log the whole betting game settles against. **Rest is measured by DISPLACEMENT, never instantaneous velocity**: three steady states read as motion forever while a body sits still — raycast suspension holds a constant +lv.y ≈ 0.73, buoyancy holds another, and a stuck car under driver control yaws in place — and the old `vmax` test converged on none of them, running 48 % of scenes to the cap. The threshold is derived, not tuned: 0.6 m of drift across a 75-tick window caps a body under 0.5 m/s, so no two bodies below it can close hard enough to log a contact (`TOUCH_DV` is 1.4). It must be that generous because `end:'coast'` cars decay asymptotically. Deliberately exempt from the gate: disturbed **props** (markets latch on first move, and a knocked cone rolls forever) and **sunk** cars. Events are fixed-order read-only scans inside the step loop; a new event kind MUST get an `EV_CODE` or it hashes as `undefined`. The FNV event hash is the determinism fingerprint (bit-identical node↔browser).

**Betting core (`markets.js` + `economy.js`) — headless-first.** `generateMarkets(scene, {labelOf})` is **outcome-blind by construction** — it never imports recorder.js; inputs are the scenario plus the static `CALIB` table of empirical hit rates keyed kind → template → bucket. Every market carries its `cal` bucket tag so the harness measures exactly what the generator prices. Odds are decimal ×100 integers, `oddsFromP = (1 − marginFor(d)) / p` clamped ×1.10–×50; margin eases 6 % → 2 % at d ≥ 8. Offering rules learned from the Monte Carlo: markets with p > 0.85 clip at the odds floor and pay the BETTOR, markets with p < 1.5 % clip at the pricing floor and pay the house — neither is offered; the headline pair is exempt (the spec requires both sides) and gets a ×1.03 floor. `settleMarket(market, rec)` is the only outcome reader.
Two calibration rules that cost real debugging:
- **`tools/calibrate.mjs` MUST pass `{ all: true }`.** Offering is itself a function of CALIB, so measuring rates over only the offered set feeds a truncated sample back in as the full-population prior. That loop does not converge: regenerating once put the house edge at +14.6 %, splicing and regenerating again flipped it to −42.2 %. Calibrating over every candidate breaks it — the table then describes the world instead of the previous table's selection.
- **There is deliberately no difficulty axis on CALIB.** When the high bands ran under target, two corrections were built and both made the gate WORSE (d8–10 to −7.7 %, then −4.5 %, from −2.8 %). A marginal per-(kind, difficulty) rate double-counts scene composition CALIB already prices via role and template; an observed/expected ratio avoids that but could not be estimated better than ±0.06 from a 400-scene sample — larger than the effect. The real cause was a stale table measured against a build where cars fell through bridges. **Regenerate CALIB first**; reach for a difficulty correction only with a calibration sample bigger than the Monte Carlo validating it.

`economy.js`: injectable store (localStorage in app, memory in tests), profile v1, slip = singles + one parlay (integer stakes ≥ $1, total ≤ bankroll — the bankroll cannot go negative by construction), `placeSlip` deducts at bet time, `settleRound` pays ⌊stake·oddsH/100⌋ (stake included), pushes the ledger, advances the campaign, and handles ROCK BOTTOM. **Achievements** (`achievements.js`) evaluate only on money rounds — Exhibition seeds are re-runnable, so unlocking there would let a player farm every badge off one seed — and unlocks are append-only and idempotent. The **daily seed** is deliberately not a special case in the money path: a normal money round with a date-derived seed (no profile key, so everyone shares the scene), where the settled-seed ledger already enforces one attempt. Streak arithmetic parses the key at **midday** so a DST shift cannot move "yesterday" by a day.

**Game UI (`betui.js` + `povcam.js`).** `betui.js` owns every pixel of the betting layer and NO sim state — main.js drives the lifecycle and calls `openRound / setPhase / tickLive / settle / closeRound`; the only callbacks out are `onLock` and `onNext`, and audio is **injected** rather than imported so the module stays mountable with no AudioContext. **Live chips are a preview, not settlement**: `buildLiveIndex` precomputes per market the tick it becomes definitively won and definitively lost, so a chip flips the frame the player sees it; by construction the state at `restTick` equals `settleMarket`, and `settle()` overwrites every chip from the authoritative report regardless. Betting stays open through the freeze — **time restarting is the lock**. Bugs worth not reintroducing: re-rendering `#mklist` innerHTML on a tap resets scrollTop and throws the player to the top (tap path uses `syncMarketRows`); **there is deliberately no tap-anywhere-to-resume** — the freeze exists to study the scene and finish a slip, so a global pointerdown handler meant every tap aimed at a market, the crosshair or the camera restarted time instead; `#fzGo` and the space key are the only exits, and the freeze layer is a pass-through box (`pointer-events: none`, children re-enabled) so it never covers the scene or eats a click; a half-built parlay sits at one leg with a hint and blocks PLACE rather than silently dragging in another pick.
`povcam.js` builds the deterministic camera rig (`'pov:'+seed`): dashcams on flagged cars (the aggressor rarely gets one — riding it hands over the read), CCTV pole, chopper orbit, witness tripod; `povBudget(d)` prunes coverage and everything unlocks at `phase === 'done'`. Camera MOTION lives in povcam.js, the LENS treatment (grain, vignettes, letterbox) is CSS on `#povfx`.

**Freeze scrub — the filmstrip (`buildStrip`/`poseStrip`/`seekPreview` in main.js).** There is no rewind in a rigid-body world, so a backward seek used to `sim.reset()` and replay from t0 — correct, but ~30 ms of mesh rebuild plus up to 600 steps *per drag frame*, which crawled under the thumb. Entering the freeze now captures the last 10 s of **visual** state once (`buildStrip`) and a seek becomes a pure pose: **0.016 ms**, no physics, no rebuild, no allocation. Captured per tick is exactly what `syncVisuals` interpolates — chassis pose, steer, suspension travel, wheel spin, plus dynamic props; debris is not captured because nothing may collide before the incident tick. `poseStrip` writes `prev` alongside `cur` so the alpha lerp is a no-op while frozen and the first stepped frame after resume interpolates from what is actually on screen.
The payoff is structural, not just speed: **the sim is never rewound at all** — it stays parked on the incident tick, so `resumeRound` cannot hand the player a scene the odds were never priced on (verified: `hashState()` is bit-identical across every seek, and a resumed round reaches the recorder's exact `restTick`). Capture order still matters — `buildStrip` calls `reset()`, so fx must `detachSim` first, `targetMap` must be rebuilt from the fresh meshes, and the six hooks are nulled for the capture pass then restored by `attach` + `hookRoundCinematics`. The old re-sim path survives in `seekPreview` as the fallback for a seek with no strip, and `round.seekTo` still exists to throttle *that* path to one rebuild per frame.

**Loading + motion.** `makeProgress` is ONE component reused three ways (boot preload · per-round pre-sim beat · scene loading) against shared `.pbar`/`.pfill` markup. **Reduced motion is a live setting, not a boot constant**: the OS media query seeds it and the in-game toggle can turn it ON, but the toggle never overrides an explicit system preference back to "full". It mirrors onto `body.reduced-motion` so CSS, the camera tween and the bankroll count-up honour one switch.

**`main.js`** owns the shell: renderer (ACES, PMREM RoomEnvironment, shadow frustum refit per scene size), camera fit/tween, showroom builder (flow layout, sections back→front: scenery → props/roads → vehicles), menu/pause/fullscreen/orientation, settings, boot, and the round lifecycle. **Cameras**: Orbit (default) and freecam — pointer-lock + WASD on desktop, fixed joystick + look-drag + ▲▼ on touch. Critical: **never run `controls.update()` while freecam or a scripted POV owns the camera** — OrbitControls re-aims at its target even when `enabled=false`. Crosshair targeting uses a per-round `Object3D → group id` map (rebuilt per round), raycast from screen centre in freecam or the pointer in orbit.
**Fog scale follows the LIVE camera during a round** (`syncFogScale`), not the distance `fitCamera` happened to pick: the metric is `fogBox.distanceToPoint(camera.position) / 8` clamped 1–20, against the round's bounding box cached at fit time. Distance to the box, **not to its centre** — that was the first version and it is wrong on exactly the scenes that matter, since a dashcam on a car at the far end of a 150 m highway is 120 m from the centroid while sitting 2 m from what it is filming, so it inherited the wide shot's ×20 and had its weather deleted. `distanceToPoint` is 0 anywhere inside the scene, so any camera among the cars gets weather at full strength. This is also what keeps weather **fair**: backing off far enough to read the whole scene is the same motion that pushes the fog away, so the wide read stays clear at every weather. `fitCamera` still seeds the scale from `dist` (where the camera is *going*) because mid-tween the camera is still back at the last shot. Rounds only — the showroom's free camera would otherwise thin its fog just by flying away from the display floor.

**Conventions everything relies on:**
- Vehicle space: forward = **+X**, ground = **y 0**, width along z. Builders return a `THREE.Group`; `buildVehicle()` recenters on x/z.
- Determinism: same seed+type+paint ⇒ identical vehicle. Type choice, build randomness and naming each use a *separately derived* rng (`makeRng('t:'+seed)`, `'b:'+seed+':'+typeId`, `'n:'+…`) so forcing a type doesn't shift the build stream. Never call `Math.random()` inside builders — always the passed `r`.
- **The geometry kit (`lib.js`)** produces the look; prefer it over raw boxes. `slab(mat, {…})` is a chamfered-box frustum carrying `userData.pt`; `faceQuad(pt, side)` + `subQuad` + `quadPrism`/`panesOnQuad` place thin plates flush on any face including sloped ones (all windows, lights and grilles work this way). `faceQuad` assumes slab-shaped `pt` — on `wedge()` (plan-tapered) faces the panes skew, so use round `cyl` lamps there. `hexa(bottom4, top4)` is the underlying 8-corner solid; winding order is exact — copy an existing call rather than deriving anew. All geometry is non-indexed + `computeVertexNormals()`; materials come from the per-build `matFactory()` which dedupes by parameter key.
- **Family builders (`families.js`)** — `car()`, `truckFront()`+`chassis()`, `van()`, `bus()` — are config-driven; most of the 92 registry entries are parameter sets. Only genuinely novel shapes (F1, dragster, excavator, tram, bikes) get bespoke builders.
- **Registry (`vehicles.js` `REG`)**: `{ id, label, cat, build(r, M, ctx) }`. `ctx.paint` must win over any `paintHex` default. Adding a REG entry updates the UI, counts and smoke test automatically; optionally add a flavor name in `names.js`.

## Product rules (do not quietly change these)

- **Money**: integer dollars only; min bet $1; max = bankroll; payout = ⌊stake × odds⌋. Singles ×1.10–×50, parlay capped ×500. House margin ~6 % easing to ~2 % at d ≥ 8 — hard reads pay nearly fair. **Never move `marginFor` to make a gate pass**; that is changing the product to fit the test.
- **Anti-cheese**: campaign seeds hidden until settlement; each seed settles winnings **once per profile** (ledger); custom, shared and replayed seeds always run as **Exhibition** — playable, zero bankroll/ledger/stats/achievement mutation.
- **Difficulty** controls incident subtlety, cast size and similarity, camera coverage, the freeze (removed at d ≥ 8), near-miss (≥5) / decoy (≥6) / multi-incident (≥7) rates, and odds spread. Never unfair randomness: every outcome must be readable in the preview if you look at the right thing — d controls how hard "the right thing" is to spot. Campaign draws d on a geometric rarity curve (`P(d) ∝ 0.62^(d−1)`).
- **Persistence**: `localStorage["crashbet.profile.v1"]`, versioned JSON — bankroll, campaign cursor, current round + slip draft, settled-seed ledger (ring buffer, 500), stats, achievements, daily streak, settings. Boot always resumes the last unfinished round. Reset wipes after a confirm.
- Vehicles and property only. No people, no gore, ever.

## Gotchas learned the hard way

- **Color math must stay in sRGB**: pass `THREE.SRGBColorSpace` to `getHSL`/`setHSL`. Linear-space HSL turns saturated reds salmon.
- Look-tuning baseline (don't casually change): exposure 1.18, key 1.7, hemi 0.55, fill 0.45; glass `#1b2836` rough 0.32 env 0.85. Brighter key lights wash colors through ACES. Weather multiplies these; it never replaces them, so a clear scene renders the baseline exactly.
- **You cannot darken a scene with the three lights alone.** `scene.environment` (PMREM RoomEnvironment) is a constant image-based flood that ignores every light intensity — the knob is `scene.environmentIntensity`. Symptom: killing the key to near zero yields a bright scene with black shadows instead of a dark one.
- `Mesh.position` is a read-only accessor — never `Object.assign(mesh, { position: … })`.
- Tall truck beds must pass `stacks: false` to `rigidTruck` or the exhaust stacks clip through the bed.
- Old vehicles must be disposed via `disposeGroup()` before replacement.
- `run.bat`'s port cleanup uses `findstr /c:` deliberately: a bare space in a `findstr` pattern means OR and would kill far too much.
- Physics wheel discovery keys off `userData.wheel` set in `parts.js` `wheel()` — a bespoke builder rolling its own wheel cylinders gets no suspension. Prefer `P.wheel`/`P.axle`.
- Prop bodies whose `node` IS the prop group must treat the node-local pose as identity in `_addPropRig` — reading `node.position` after the group was placed double-applies the offset (ramp colliders were 2× off).
- Rapier objects need explicit cleanup: `CrashSim.dispose()` frees world, event queue and controllers. Never drop a sim without it (wasm leaks silently).
- **A pinned scenario that bypasses the generator does not prove the generator works.** The causeway shipped a bridge over dry land for an entire phase because `generateScene` enumerated `world` keys and dropped `water`, while the `water` pin hand-wrote `world.water`. Likewise the `bridge` pin never caught the missing deck surface because its cars enter where deck y is exactly 0. `generateScene` must SPREAD `topo.world`, never re-list its keys.
- **Never compare crash-mode replays after N wall-clock frames — compare at the same TICK.** Auto slow-mo is on a 5-second real-time cooldown, so the same frame count executes different tick counts and hashes differ for reasons unrelated to determinism.
- **Never animate `opacity` from 0 with `animation-fill-mode: both` on anything load-bearing.** An animation that never starts holds its 0 % keyframe forever. Animate `transform` instead: a frozen scale degrades to "slightly small", a frozen opacity degrades to "the content does not exist".
- **`.mbtn` sets `width: 100%`, so it cannot go into a flex row unchanged** — `flex: 0 0 auto` resolves the basis to that 100 % and the first button eats the row. Add `width: auto` when flexing it.
- **A calibration correction must be measured as observed/expected against the model's own predictions**, never as a marginal rate — a marginal rate re-counts whatever the model's other keys already explain. And if the 95 % interval on your calibration sample is wider than the effect, the sample is too small to act on.
- The embedded Browser pane reports `document.hidden`, never fires `requestAnimationFrame`, and freezes CSS transitions and animations. Drive the game with `__app.pump(now)`; a "wrong" `getBoundingClientRect` may be a transition frozen mid-flight; and `computer{action:screenshot}` may time out (rasterize via `foreignObject` + a local sink instead). Do NOT conclude the sim/camera/HUD is broken because nothing moves on its own.
- **In that pane the `ResizeObserver` on `#stage` never fires either**, because observer callbacks are delivered as part of the rendering steps the pane does not run. So `resize()` only ever runs once, at boot, when `#stage` can still be 0×0 — and the canvas stays 0×0 while everything else works perfectly: the sim steps, `renderer.info` counts draw calls, `pump` returns true, and `toDataURL` hands back `"data:,"`. If a screenshot comes back blank, check `renderer.domElement.width` before suspecting the scene, then re-run what `resize()` does: `renderer.setSize(stage.clientWidth, stage.clientHeight)` + `camera.aspect` + `updateProjectionMatrix()`.
- **CSS edits need a hard reload in the preview pane** — the stylesheet is cached, so re-measuring right after an edit silently reports the OLD layout.
- `?crash=N` boots with no user gesture, so WebAudio stays suspended until the first pointerdown — not a bug.
- `.claude/launch.json` uses `autoPort: true` (5183 is often held) — read the port from `preview_start`'s result rather than hardcoding it. `run.bat` still pins 5183 for manual use.
- **Never push unless explicitly told.**
